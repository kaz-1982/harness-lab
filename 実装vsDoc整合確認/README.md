# 実装↔ドキュメント整合ハーネス v0.1

先に動いている**実装(コード)**と、**設計ドキュメント**の乖離(ドリフト)を、**コードを正**として解消する案件非依存ハーネス。**入力に2つのディレクトリを指定**して使う。

## いつ使うか

- アプリを作り終えた後に改善を重ね、設計ドキュメントが古くなった
- レビュー/引き継ぎ前に「設計docと現コードを一致させたい」
- 別チーム/別ツールが実装し、設計に反映されていない差分がある

> 「設計 → 実装」の前方向きにドキュメントを作る生成ハーネス(`開発doc作成_v*` 等)の**後工程**。生成ハーネスの中には入れず、これ単体で回す。

## 使い方

1. **2つのディレクトリを指定する**(どちらかで):
   - **`input/config.md` に記入**(推奨・再現性あり): `APP_DIR` と `DOC_DIR` を絶対パスで書く。
   - または起動後に**会話で伝える**(config より優先): 「APP_DIR=… / DOC_DIR=… の整合を取って」
2. このディレクトリで Claude Code(Opus 推奨)を起動し、「整合を取って(コードを正に)」と伝える。

3. Claude が R1〜R5 を実行:
   - **R1 トリアージ** どのdocを直すか仕分け(`harness/01_triage.md`)
   - **R2 ドリフト検出** ID単位の差分表 `_drift_report.md` を生成
   - **R3 影響波及** 直すdocを確定
   - **R4 修正** コードを正に1docずつ追従
   - **R5 検証** 機械チェック緑 + 差分の全決着 + 対応ダイジェスト再生成

任意で `SCOPE`(対象の限定)、`OUT`(レポート出力先)も指定可。詳細は `harness/00_inputs.md`。

## 構成

```
実装doc整合ハーネス_v01/
├── CLAUDE.md                    # 運用ルール + 起動時の挙動(必読)
├── README.md                    # このファイル
├── ChangeLog.md
├── input/
│   └── config.md                # ★ APP_DIR / DOC_DIR を記入する設定ファイル
├── harness/
│   ├── 00_inputs.md             # 2ディレクトリ入力の契約・足場
│   ├── 01_triage.md             # doc区分マトリクス(修正必要度)
│   ├── 02_workflow.md           # Step R1〜R5(R2/R4 に並列ファンアウト・レシピ)
│   ├── 03_quality_checklist.md  # 手動観点
│   ├── agents/                  # 並行サブエージェントの役割定義 + 並列の制約
│   └── templates/_drift_report.md
└── runs/                        # 実行ごとの控え(正本は DOC_DIR 側)
```

## 設計メモ

- **差分の正本は DOC_DIR 側**(`_drift_report.md` を DOC_DIR に置く)。このハーネスの `runs/<案件名>/` には控えと実行メモのみ。
- DOC_DIR が ID体系(`_id_registry.md`/`check.py`)を持つ生成ハーネス成果物なら、それを基準に突合し `check.py` 緑を合格条件にする。素のドキュメント群なら章/見出し単位で突合する(`harness/00_inputs.md` 参照)。
- `APP_DIR` に「コード↔設計 対応ダイジェスト」(例 `DESIGN_DIGEST.md`)があると検出が速い。無ければ R2 で簡易版を作る。

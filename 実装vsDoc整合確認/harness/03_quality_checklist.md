# 品質・整合性チェック観点(手動)

機械チェック(`check.py` 等)は形式・参照・整合しか見ない。以下は**機械では測れない意味の妥当性**を人手(Claude)で確認する観点。R5 で実施し `_review_log.md`(あれば)に記録する。

## ドリフト検出の網羅性(R2)

- [ ] APP_DIR のエンドポイント / テーブル / 主要モジュール / テストを**漏れなく**棚卸ししたか(一部だけ見て「整合済」と誤認していないか)
- [ ] `git log` / `ChangeLog` の各変更が、対応するdoc差分として `_drift_report.md` に現れているか
- [ ] 「追加」だけでなく「削除(docにあるが実装に無い)」も探したか

## 根拠の確かさ(R4)

- [ ] 断定した値・enum・パス・HTTPステータス・制約は、**実コード/実スキーマで確認**したものか(ダイジェストや記憶だけで書いていないか)
- [ ] 確認できなかった点は `TBD` として残したか(推測で埋めていないか)

## 修正の質

- [ ] 一致していた設計を**むやみに書き換えていない**か(差分は最小か)
- [ ] テンプレ構造・frontmatter・節立てを壊していないか
- [ ] 新規IDの採番に穴/重複が無いか。ID台帳と本文参照が一致するか
- [ ] 用語の表記揺れを持ち込んでいないか

## 切り分け

- [ ] 「コードが正しい改善」と「設計意図から外れたバグ」を区別したか。後者を doc に追従させて**バグを仕様として固定**していないか
- [ ] 方式差を、影響範囲を確認せずに勝手に全面改訂していないか

## 決着

- [ ] `_drift_report.md` の全乖離が「修正済 / 据え置き(理由) / コード側課題」のいずれかになっているか(未決着が残っていないか)
- [ ] 「直さなかった(整合確認済)」記録を残したか

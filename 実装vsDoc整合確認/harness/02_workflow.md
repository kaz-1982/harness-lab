# ワークフロー(Step R1〜R5)

入力 `APP_DIR`(実装) + `DOC_DIR`(設計ドキュメント)を受け取り、**コードを正**としてドキュメントを追従させる。入力契約は [00_inputs.md](00_inputs.md)、トリアージ表は [01_triage.md](01_triage.md)。

---

## R0. 入力確定(前段)

1. `APP_DIR` / `DOC_DIR` を特定(不明なら質問)。
2. 足場(`DESIGN_DIGEST.md`・`_id_registry.md`・`_doc_plan.md`・`check.py`・両者の `git log`/`ChangeLog`)の有無を確認。
3. 正の原則(既定=コードを正)をユーザーと合意。

---

## R1. トリアージ(直す範囲を決める)

[01_triage.md](01_triage.md) の区分マトリクスで `DOC_DIR` のドキュメントを仕分ける。`SCOPE` 指定があれば併せて絞る。**据え置き区分は触らない**ことをこの時点で確定する。

---

## R2. ドリフト検出(差分の洗い出し)

差分を**勘でなく機械的に**出し、`_drift_report.md`(テンプレ `harness/templates/_drift_report.md`)に **ID単位(または章/見出し単位)の差分表**としてまとめる。入力ソースは4系統:

1. **コード/スキーマ** = `APP_DIR`: API エンドポイント・DBスキーマ(実DBの `.schema` 含む)・主要モジュール・テストを棚卸し
2. **対応ダイジェスト** = `APP_DIR/DESIGN_DIGEST.md` 等(あれば)
3. **変更ログ** = `git -C APP_DIR log` + `DOC_DIR/ChangeLog.md`
4. **ID台帳 / doc現状** = `DOC_DIR/_id_registry.md`(あれば)と各ドキュメント本文

各差分に**乖離種別**(01_triage.md)を付ける。`_drift_report.md` には少なくとも: ①サマリ(乖離件数・修正対象doc・要ユーザー判断) ②ID差分表 ③doc別修正方針 ④要ユーザー判断 ⑤**整合確認済(修正不要)** を記す。

### 並行サブエージェントで高速化(ID種別ファンアウト)

棚卸し〜検出はサブエージェントの並列実行で大きく短縮できる。役割定義は [agents/](agents/) を参照。

1. **棚卸し(任意)**: `code-inventory`(実装) と `doc-inventory`(doc) を**並列**で起動し、それぞれ構造化インベントリを返させる。小規模なら省略し検出から始めてよい。
2. **検出ファンアウト**: `drift-detector` を **ID種別ごとに並列**で起動する。境界の例:

   | サブ | 担当ID | 突合対象(典型) |
   |---|---|---|
   | API担当 | `A-*` | 実エンドポイント ↔ API仕様/エラー |
   | データ担当 | `T-*`/`E-*` | 実DBスキーマ ↔ 物理データ/テーブル定義/DDL/論理ER |
   | 機能担当 | `F-*` | コード/git ↔ 機能一覧 |
   | テスト担当 | `UT-*`/`AT-*` | 実テスト ↔ テスト仕様 |

   各サブは**読み取り専用・担当ID範囲のみ**。戻りの差分表を**親がマージ**して `_drift_report.md` を確定する。
3. **守る制約**(暴走防止 / [agents/README.md](agents/README.md)):
   - 検出系は read-only なので何体でも安全に並列。
   - 各サブは**自己完結プロンプト**(APP_DIR/DOC_DIR/担当ID/コードを正/推測禁止)で渡す。
   - `_id_registry.md` 等の台帳更新・**ユーザー判断(方式差)**・マージは**親が集約**。

---

## R3. 影響波及トレース

各乖離について、`_id_registry.md` と各docの「関連トレーサビリティ」を辿り、**実際に手を入れるドキュメントの確定リスト**を作る。R1 のトリアージと突き合わせ、「修正必須 × 波及あり」を最終スコープにする。1つの実装変更が複数docへ波及することを取りこぼさない(例: 新APIは API仕様 / エラー / 状態遷移 / 受け入れテストへ)。

**方式差**で要件・基本設計の前提を覆す規模なら、ここで一旦止めてユーザーに方針確認(全面改訂 / 方式注記で吸収 / 併記)。

---

## R4. 修正実行(1ドキュメントずつ)

確定スコープを **1つずつ**、以下のサイクルで直す:

1. **該当doc + 根拠コードを読む**。断定する値・enum・パス・ステータスは**実コードで確認**(推測しない)。
2. **最小差分で追従**。テンプレ構造・節立て・frontmatter を壊さない。一致していた箇所は触らない。
3. **ID/用語の更新**: 逆同期で増えた機能/API/テーブル/テスト等の**新規IDを `_id_registry.md` に採番登録**(ID体系がある場合)。用語集があれば突合。
4. **版を上げて記録**: 各docの改訂履歴に「実装整合(逆同期)」の行を追加し、レビュー状態の整合チェック日付・承認行を更新。`_drift_report.md` の当該行を「修正済」に。

> **整合確認のみで修正不要だったdoc**(コードと一致していた設計)も、`_drift_report.md §整合確認済` と `_review_log.md` に「整合確認済」として残す。**直さなかったことを記録に残す**のが目的。

### 並行サブエージェントで高速化(doc単位ファンアウト)

確定スコープが複数docに渡るなら、`doc-reconciler` を **担当docごと(重複なし)に並列**で起動する([agents/doc-reconciler.md](agents/doc-reconciler.md))。

- **割当**: 独立したdocを1サブ1〜数ファイルで割り当てる。**同じファイルを2サブに割らない**(書き込み衝突防止)。関連の強いdoc(例: 物理データ+DDL+テーブル定義)は同じサブにまとめると一貫性が出る。
- **渡すもの**: 担当docパス + `_drift_report.md` の該当行 + **確認済みの実装事実**(値・enum・パス・ステータス)。サブは記憶を共有しないので毎回同梱する。
- **親だけがやる**: 新規IDの**一括採番**(`_id_registry.md`)、`_doc_plan.md`/`_review_log.md` の更新、検証(R5)。サブにはこれらを触らせない。
- サブの戻り(変更要約 + 新規採番候補ID)を受け、親が台帳に反映する。

> 採番の順序: サブが「新規採番候補ID」を返す → 親がまとめて `_id_registry.md` に登録 → R5 の `check.py` でID参照の整合を確認、という流れにすると衝突しない。

## R5. 検証・締め

- [ ] (ID体系あり)`DOC_DIR` の機械チェック(`python3 {DOC_DIR}/harness/tools/check.py` 等)が緑
- [ ] `_drift_report.md` の全乖離が「修正済 / 据え置き(理由付き) / コード側課題」のいずれかで決着
- [ ] `_doc_plan.md` 等の版・状態が更新済
- [ ] [03_quality_checklist.md](03_quality_checklist.md) の手動観点を確認
- [ ] `APP_DIR/DESIGN_DIGEST.md` 等の対応ダイジェストを**再生成**し、コードと設計が再一致することを確認
- [ ] `DOC_DIR/_review_log.md`(あれば)末尾に「逆同期 完了」を日付付きで記録。`runs/<案件名>/` に `_drift_report.md` の控えを残す

---

## 緊急停止条件

- **方式差**が要件・基本設計の前提を覆す規模だった → 修正前にユーザーへ方針確認
- コードがバグで設計が正しい可能性がある乖離 → doc を直さず「コード側課題」として切り分け、ユーザー判断を仰ぐ
- ユーザー回答が複数解釈を持ち、推測すると後段に重大な影響がある

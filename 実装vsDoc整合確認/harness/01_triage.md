# R1 トリアージ — ドキュメント区分マトリクス

**全量は直さない。** ドキュメントを「逆同期時の修正必要度」で仕分けてから着手する。下表は一般的な設計ドキュメント体系(要件 R-* / 基本設計 B-* / 詳細設計 D-* / テスト)を想定した既定。`DOC_DIR` の体系に合わせて読み替えてよい。

| 区分 | 典型ドキュメント | 逆同期時の方針 |
|---|---|---|
| **据え置き(点情報)** | 事前検討、企画、**要件定義(業務要件・ユースケース・SRS・非機能)** | その時点の意思決定記録として保持。**要件レベルの判断が実装で変わった時のみ**巻末に追補注記 |
| **据え置き(アーキ不変)** | 基本設計の**方式系**(システム方式・アーキテクチャ・ネットワーク・方式共通・運用) | アーキテクチャ・方式は不変前提。ドリフト検出で当たりが出た時だけ追記 |
| **要確認** | **機能一覧・論理データモデル・セキュリティ設計** | 機能/データ項目/方針の増減をコードと突合 |
| **修正必須** | 詳細設計の**ロジック・IF系**(クラス図・フローチャート/PDL・モジュール仕様・API仕様・エラー/メッセージ) | コードと1対1で突合し追従(機能概要・詳細はここ) |
| **スキーマ突合で修正** | **物理データモデル・テーブル定義・DDL** | 実DBスキーマ・DDLと突合。方式差(後述)があれば要ユーザー確認 |
| **変わった分だけ修正** | **シーケンス図・状態遷移** | 処理フロー・状態が変わった図表のみ |
| **挙動変化分を追従** | **テスト仕様(単体・受け入れ)** | 実装で挙動が変わったケースを更新 |

## 乖離種別(R2 で各差分に付ける)

| 種別 | 定義 | 典型対応 |
|---|---|---|
| 追加 | 実装にあり、docに無い(新機能/新エンドポイント) | 新ID採番 + 該当docに追記 |
| 変更 | 両方にあるが内容が違う(仕様変更) | docを実装に合わせ改訂 |
| 削除 | docにあり、実装に無い(廃止) | docから除去 or 「廃止」注記 |
| **方式差** | 設計が前提した実現方式と実装方式が違う(影響大) | R3 で波及を確定し、**修正前にユーザーへ方針確認**(全面改訂 / 方式注記で吸収 / 併記) |
| 逆(未実装) | docにあるが未実装(設計だけ先行) | コード側TODO or docを「将来」に格下げ |

## 判断のコツ

- 「コードが正しい改善」と「設計意図から外れたバグ」を切り分ける。後者は doc を直さず**コード側課題として報告**する。
- ロジックの中身(評価式・アルゴリズム等)は**意外と設計どおりのことが多い**。先に突合して「一致=直さない」を確定させると、修正対象が大きく絞れる。

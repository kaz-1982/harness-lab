# 入力契約(2ディレクトリ)

このハーネスは **2つのディレクトリ**を入力に取る。起動時にこの2つを確定してから手順(`02_workflow.md`)へ進む。

## 指定方法(優先順)

1. **会話での明示指定**(最優先): 「APP_DIR=… / DOC_DIR=…」と伝えられたらそれを使う。
2. **`input/config.md`**: 記入済みなら APP_DIR / DOC_DIR / 任意項目をここから取る。リポジトリに残るので**再現性がある**(同じ案件を再度回すときに便利)。
3. どちらも無ければ**質問**する(推測しない)。

> `input/config.md` はテンプレートが用意されている。各項目の `:` の右に絶対パスを記入するだけ。

## 必須入力

| 名前 | 意味 | 例 |
|---|---|---|
| `APP_DIR` | **実装**のルート。コード・テスト・スキーマ・`git` を含む | `/path/to/MyApp` |
| `DOC_DIR` | **設計ドキュメント**のルート(または `output/` 等の本体ディレクトリ) | `/path/to/MyApp_docs` または `.../output` |

- どちらか特定できないときは**推測せず質問**する(ルール2)。
- 2つは別リポジトリ/別ディレクトリのことが多い。両方を Read できるパスで起動する。

## 任意入力

| 名前 | 意味 | 既定 |
|---|---|---|
| `SCOPE` | 対象ドキュメントの限定(例「詳細設計だけ」「API周りだけ」) | 全体(R1 トリアージで自動仕分け) |
| `OUT` | ドリフトレポートの出力先 | `DOC_DIR` 配下(`output/_drift_report.md` 等、なければ `DOC_DIR/_drift_report.md`)。控えを `runs/<案件名>/` にも置く |
| `CANONICAL` | 正の原則 | `code`(コードを正)。`per-drift` で乖離ごとに判断 |

## 足場(あれば活用 / 無くても動く)

検出精度が上がるので、起動時に存在を確認する:

1. `APP_DIR/DESIGN_DIGEST.md` 等の **コード↔設計 対応ダイジェスト**(各章「現状コードとの対応」が差分の宝庫)。無ければ R2 で簡易版を作るところから始めてよい。
2. `DOC_DIR` が **生成ハーネス(`開発doc作成_v*` 等)の成果物**の場合に存在しうるもの:
   - `_id_registry.md`(ID台帳) … 差分を ID 単位で突合する基準
   - `_doc_plan.md`(作成計画・版・状態)
   - `_review_log.md`(レビュー記録) … R5 で逆同期の記録を追記
   - `harness/tools/check.py` … R5 の機械チェックに使う
   - 各ドキュメントの frontmatter / 改訂履歴 / 「関連トレーサビリティ」
3. **両者の変更履歴**: `git -C APP_DIR log --oneline`、`DOC_DIR/ChangeLog.md`。「いつ・何を変えたか」が差分の当たり所になる。

## DOC_DIR の型による進め方の差

- **ID体系あり(生成ハーネス成果物)**: `_id_registry.md` を基準に F-/A-/E-/T-/S- 単位で突合。新規IDは台帳へ採番登録。`check.py` 緑を R5 の合格条件にする。
- **ID体系なし(素のドキュメント群)**: ファイル/章/見出し単位で突合。ID台帳の代わりに `_drift_report.md` を唯一の差分台帳とし、各docの「変更履歴」節(なければ作る)に逆同期を記録する。

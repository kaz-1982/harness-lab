# 実装↔ドキュメント整合ハーネス — 運用ルール (Claude 向け)

このリポジトリは、**先に動いている実装(コード)と、設計ドキュメントの乖離(ドリフト)を、コードを正として解消する**ための案件非依存ハーネスです。**入力に2つのディレクトリ(実装 / ドキュメント)を指定**して実行します。

このファイルは Claude Code が会話開始時に必ず読み込みます。以下のルールに従って動いてください。

> 立ち位置: これは「設計 → 実装」の前方向きにドキュメントを作る**生成ハーネス**(例: `開発doc作成_v*`)とは別物で、**実装が先に進んだ後の逆方向(実装 → 設計)の整合**を担う後工程ハーネスです。生成ハーネスの Phase 0〜4 の後に、必要なときだけ回します。

---

## このハーネスがやること(全体像)

```
入力: APP_DIR(実装) + DOC_DIR(設計ドキュメント)
  │
  R1 トリアージ   どのドキュメントを直すか仕分け(据え置き/要確認/修正必須)
  R2 ドリフト検出  コードとdocを突合し、ID単位の差分表 _drift_report.md を作る
  R3 影響波及     1つの差分が波及する先のドキュメントを確定
  R4 修正実行     コードを正にドキュメントを1つずつ追従
  R5 検証        機械チェック緑 + 差分の全決着 + 足場の再生成
  │
出力: 更新された DOC_DIR のドキュメント + _drift_report.md(差分台帳)
```

詳細手順は [harness/02_workflow.md](harness/02_workflow.md)。

---

## 起動時の挙動

ユーザーは通常「この実装とこのドキュメントの整合を取って」という形で、**2つのディレクトリ**を渡してきます。

1. **入力の確定**: `APP_DIR`(実装ルート)と `DOC_DIR`(設計ドキュメントのルート)を特定する。
   - **まず [input/config.md](input/config.md) を読む**。記入済みなら APP_DIR / DOC_DIR / 任意項目をそこから取る。
   - 会話で明示パスを伝えられた場合は**そちらを優先**(config より上位)。
   - config も会話も無く特定できないなら**推測せず質問**する。
   - 入力契約の詳細は [harness/00_inputs.md](harness/00_inputs.md)。
2. **足場の確認**: 以下があれば活用する(無くても動く)。
   - `APP_DIR/DESIGN_DIGEST.md` 等の「コード↔設計 対応ダイジェスト」
   - `DOC_DIR` 配下の `_id_registry.md` / `_doc_plan.md` / `check.py`(`開発doc作成` 系ハーネスで生成された設計の場合)
   - 両者の `git log` / `ChangeLog.md`
3. **正の原則の合意**: 既定は **コードを正(逆同期)**。乖離ごとに「コードが正しい改善か / 設計意図から外れたバグか」を切り分ける余地を残す(後者は docではなくコード側課題として報告)。
4. R1 → R5 を [harness/02_workflow.md](harness/02_workflow.md) に従って進める。各ドキュメントは**1つずつ**直す。

ユーザーから明示の指示があればそれを優先。ただし DOC_DIR が生成ハーネス成果物の場合、そのハーネスの規約(テンプレ準拠・ID台帳・整合チェック)を尊重する。

---

## 守るべきルール

1. **コードを正(ただしバグは切り分け)**: 実装(コード + テスト)を正本に doc を直す。実装側が誤りに見える乖離は、doc を直さず「コード側課題」として報告する。
2. **推測で埋めない**: 断定する数値・enum・パス・ステータス等は、必ず**実コード/実スキーマで確認**してから書く。確認できないものは `TBD` として残す。
3. **最小差分・テンプレ準拠**: DOC_DIR 側の既存のテンプレ構造・節立て・frontmatter を壊さない。一致していた箇所は変更しない(「直さなかった記録」を残す)。
4. **1ドキュメント1サイクル**: 「該当差分の反映 → ID/用語の更新 → 整合チェック → 版を上げて記録」を1つ通してから次へ。
5. **差分は台帳化**: 検出した乖離は必ず `_drift_report.md`(ID差分表)に記録し、各乖離を「修正済 / 据え置き(理由) / コード側課題」のいずれかで決着させる。
6. **影響範囲が要件・アーキの前提を覆す乖離(方式差)**は、修正前にユーザーへ方針確認(全面改訂 / 方式注記で吸収 / 併記)。

## 主要ファイル

| パス | 役割 |
|---|---|
| `input/config.md` | **入力設定**。APP_DIR / DOC_DIR(+任意項目)を記入する。起動時にまず読む |
| `harness/00_inputs.md` | 入力契約(APP_DIR / DOC_DIR の指定・前提・足場) |
| `harness/01_triage.md` | ドキュメント区分マトリクス(逆同期時の修正必要度) |
| `harness/02_workflow.md` | Step R1〜R5 の進行手順 |
| `harness/03_quality_checklist.md` | 手動観点チェック |
| `harness/agents/` | 並行サブエージェントの役割定義(code-inventory / doc-inventory / drift-detector / doc-reconciler)+ 並列の制約 |
| `harness/templates/_drift_report.md` | ドリフトレポート雛形(ID差分表) |
| `runs/<案件名>/` | 実行ごとの控え(差分レポートのコピー・実行メモ)。正本は DOC_DIR 側に置く |
| `ChangeLog.md` | 本ハーネス自身の改修記録 |

---

## 運用モデル

このハーネスは **Opus(`claude-opus-4-8`)での運用を前提**に設計されています。`APP_DIR` と `DOC_DIR` は**別リポジトリ/別ディレクトリ**であることが多いので、両方を読める作業ディレクトリ(共通の親など)で起動するか、絶対パスで指定してください。

## バージョン

実装↔ドキュメント整合ハーネス v0.3。2ディレクトリ入力(実装 / 設計doc)でコードを正に逆同期する後工程ハーネス。入力は `input/config.md` で指定(会話指定が優先)。並行サブエージェント方式(`harness/agents/`)で検出を ID種別・修正を doc単位にファンアウトして高速化。

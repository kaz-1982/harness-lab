# 品質チェックリスト

ドキュメント作成完了直前に必ず実行。チェック結果は `output/_review_log.md` に追記すること。

> **2層構成で運用する**。まず機械チェック `python3 harness/tools/check.py` を実行し、**自動検証できる項目**(構造・ID整合・状態突合)を緑にする。その後、本チェックリストの**手動観点**(意味の妥当性など機械では測れない項目)を人/AI が確認する。
>
> - 🔧 **機械**(check.py が判定): 1-1 / 1-7 / 2-1 / 2-2 の一部(RTM のID整合)
> - 👁 **手動**(AI・人が判断): 1-2〜1-6 / 1-8〜1-9 / 2-3〜2-7 / 各フェーズ移行の意味的判断 / Phase 3.5 ゲートの妥当性

## 2. 整合性(ドキュメント横断)

### 2-1. ID整合
- [ ] `output/_id_registry.md` と各ドキュメントのID参照が一致

- [ ] 業務要件 → 機能要件 → 設計 → テスト の対応が `output/横断/R-14_RTM.md` に登録済(RTM 作成対象の場合)
- [ ] 「孤立した機能」(対応する業務要件がない機能)がない

- [ ] **(TDD採用時)** 全 AC(R-7/R-8 の受け入れ基準)に対応する受け入れテストケース(AT-*)が存在する

### 2-3. 用語統一
- [ ] 用語集にない用語が登場していない
- [ ] 同じ概念に対する表記揺れがない(例: 「顧客」と「お客様」が混在していない)
- [ ] 略語は初出時に必ずフルスペルを併記

### 2-4. データ整合
- [ ] R-4 概念データモデル → B-12 論理ER図 → D-8 物理ER図 → D-10 DDL でエンティティ・属性が一致
- [ ] B-14 CRUD図(作成対象の場合)で、全エンティティが少なくとも1つの C を持つ機能から作られる
- [ ] B-12 論理ER図のエンティティ名と R-13 用語集が一致
- [ ] **(DDD採用時)** DM-1 ドメインモデルのエンティティ(E-)が B-12 / B-13 と同名・同IDで一致(二重定義がない)
- [ ] **(DDD採用時)** DM-1 の集約(AG-)・値オブジェクト(VO-)が `_id_registry.md` に登録され、ユビキタス言語が R-13 用語集と一致

### 2-5. UI整合
- [ ] B-7 画面遷移図に登場する画面と B-8 画面設計書が1対1対応
- [ ] 画面の操作 → API → 機能 の対応関係がトレースできる

### 2-6. API 整合
- [ ] R-10 外部IF要件 → B-11 外部IF設計書 → D-13 API詳細仕様書 で連携先・データが一致

### 2-7. 非機能 整合
- [ ] R-9 非機能要件 → B-1 システム方式 / B-19 セキュリティ で実現方法が示されている
- [ ] 性能要件(応答時間、同時ユーザー数等)に対するアーキテクチャ判断の根拠がある

## 5. チェック実行ログのフォーマット

`output/_review_log.md` への記録例:

```markdown
## 2026-04-26 R-1_業務要件定義書 単体品質チェック
- 結果: 一部要修正
- 1-1: OK
- 1-2: OK
- 1-3: OK
- 1-4: 要修正(KPI の数値目標が定性表現のまま)
- 1-5: OK
- 1-6: OK
- 1-7: OK
- 対応: ユーザーに「月次決算の所要日数の数値目標」を確認 → 「1営業日以内」と確定 → 修正済
- 整合性チェック: 該当なし(他ドキュメント未作成)
- 最終結果: OK
```

整合性チェックも同様の形式で記録。チェック対象ドキュメントの粒度ではなく、**チェック実施タイミング** ごとに1ブロック追記する形でよい。

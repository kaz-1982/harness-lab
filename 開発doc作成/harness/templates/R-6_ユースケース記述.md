# {{案件名}} - ユースケース記述

## 1. 目的
ユースケース図の各ユースケースについて、処理の流れを自然言語で詳細化する。
機能要件(R-8)・受け入れテスト(TS-1)の根拠とする。

## 2. 適用範囲・スコープ
- 含むもの: 各ユースケースの概要、アクター、事前 / 事後条件、主シナリオ、代替・例外シナリオ、頻度・優先度
- 含まないもの: 画面のレイアウト(B-7 / B-8)、内部処理ロジック(D-* 詳細設計)

## 3. 前提・依存
- 参照ドキュメント: R-5(ユースケース図)
- 採番済ID: 対応する機能 `F-*`(R-8 / B-6)があれば併記する。本書はユースケース名で識別する

## 4. 本文

### 4.1 ユースケース記述一覧
ユースケースごとに以下のフォーマットで記述する(1ユースケース=1ブロック)。

| 項目 | 内容 |
|---|---|
| ユースケース名 | {{注文する}} |
| 概要 | {{カートの商品を確定し注文を登録する}} |
| 主アクター | {{顧客}} |
| 優先度 / 頻度 | {{Must / 1日平均500件}} |

### 4.2 主シナリオ(基本フロー)
正常に完了する標準的な流れを番号付きで記述する。

### 4.3 代替・例外シナリオ
分岐・エラー時の流れを「{{ステップ番号}}a」形式で記述する。

### 4.4 事前条件・事後条件
- 事前条件: {{ユースケース開始前に満たされているべき状態}}
- 事後条件: {{ユースケース完了後に成立する状態}}

### 4.5 頻度・優先度
実行頻度と優先度(MoSCoW 等)を整理し、設計・テストの重み付けに使う。

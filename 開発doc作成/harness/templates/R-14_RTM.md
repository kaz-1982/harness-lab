# {{案件名}} - 要求トレーサビリティマトリクス(RTM)

## 1. 目的
要求が要件 → 設計 → 実装 → テストへとどう具現化されたかを **追跡可能** にする。
変更時の影響分析、監査対応、品質保証(孤立要件・孤立設計の検出)に有用。
TDD 採用時は Phase 3.5 でテストID列(UT-* / AT-*)を充填する。

## 2. 適用範囲・スコープ
- 含むもの: {{業務要件 → 機能要件 → 機能 → 画面 / API → モジュール → テストケース の対応}}
- 含まないもの: {{プロジェクト管理レベルの WBS(P-3 プロジェクト計画書で扱う)}}

## 3. 前提・依存
- 参照ドキュメント: R-1(R-B-*), R-8(R-F-*, F-*), B-6(F-*), B-7/B-8(S-*), D-13(A-*), D-12(モジュール), D-15(UT-*), TS-1(AT-*)
- 採番済ID: 本書は採番せず、各起源ドキュメントの ID を引用する
- 更新頻度: 各フェーズ完了時 / TDD採用時は Phase 3.5 でテストID列を必ず充填

## 4. 本文

### 4.1 トレーサビリティマトリクス
**列の意味**:
- 業務要件 R-B-*: R-1 で採番
- 機能要件 R-F-*: R-8 で採番(アジャイル運用時は US-* で代替)
- 機能 F-*: R-8 / B-6 で採番
- 画面 S-*: ClaudeDesign 側で採番(B-7 / B-8)
- API A-*: B-11 / D-13 で採番
- モジュール: D-12 で定義
- 単体テスト UT-*: D-15 / Phase 3.5 で採番
- 受け入れテスト AT-*: TS-1 / Phase 3.5 で採番

| 業務要件 | 機能要件 | 機能 | 画面 | API | モジュール | 単体テスト | 受入テスト |
|---|---|---|---|---|---|---|---|
| R-B-001 | R-F-001 | F-001 | S-001 | A-001 | UserService.register | UT-001 | AT-001 |
| R-B-002 | R-F-002 | F-002 | S-002 | A-002 | UserService.login | UT-005 | AT-003 |

> 1要件が複数機能に展開される場合は、行を分けて記入する(例: R-B-001 が F-001 と F-002 に展開されるなら2行)。

### 4.2 孤立要件・孤立設計のチェック
本表は機械チェック(`harness/tools/check.py` の孤立検出)の入力となる。次の状態は**問題**:

- **孤立要件**: 業務要件 R-B-* に対応する機能 F-* が無い(要件が設計に降りていない)
- **孤立機能**: 機能 F-* に対応する業務要件 R-B-* が無い(根拠不明の機能)
- **テスト未割当(TDD採用時)**: F-* / AC に対応する UT-* / AT-* が無い

検出された未解決事項は「7. レビュー状態」に記録する。

### 4.3 RTM の更新タイミング
- Phase 1 完了時: 業務要件 / 機能要件 / 機能 列を埋める
- Phase 2 完了時: 画面 列を埋める
- Phase 3 完了時: API / モジュール 列を埋める
- Phase 3.5 完了時(TDD採用時): 単体テスト / 受入テスト 列を埋める

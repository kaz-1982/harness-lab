## 設計ドキュメント連携(重要)

### 1. `docs/design/` は読み取り専用スナップショット

`docs/design/` は **設計 repo のスナップショット**です。**このディレクトリ配下を直接編集しないでください。**

- submodule 運用なら、設計 repo そのものがネストしています。実装 repo 側のコミットで中身を変えてはいけません。
- 単純コピー運用なら、次回 `sync-to-impl.sh` 実行時に上書きされるため、編集しても失われます。
- 編集が必要に見えたら、それは「設計を変えたい」サインです → 下記 2 へ。

### 2. 設計を変えたくなったら ADR を切る

実装中に「設計と違う作りにしたい / 設計の前提が崩れた」と判明したら、**影響レベルを判断**してから動きます。

| レベル | 判断 | アクション |
|---|---|---|
| **小** | 設計に影響しない実装詳細 | 実装 repo 内で対応。ADR 不要 |
| **中** | 設計に影響する(構造・IF・データが変わる) | **ADR を切る**(`origin: implementation`)→ 設計 repo に逆同期して D-* / B-* を更新 |
| **大** | 要件レベルの問題 | **実装を一時停止**し、設計 repo に戻って R-* / B-* から見直す |

ADR は両 repo の **同期キー**です。実装フェーズで生まれた決定は `origin: implementation` で記録し、設計 repo の `output/横断/ADR/` と `output/_handoff_to_implementation/_sync_log.md` に反映します。

### 3. 同期方式

この案件は以下のいずれかで設計 repo と同期します(どちらか一方に ✅):

- [ ] **git submodule**: `docs/design/` が設計 repo の submodule。更新は `git submodule update --remote docs/design`。
- [ ] **単純コピー**: 設計 repo 側で `bash harness/tools/sync-to-impl.sh <この実装repoパス>` を実行して取り込む。

同期したら、設計 repo の `_sync_log.md` に commit / 日付 / 同期した ADR を記録すること。

### 4. TDD 採用案件の場合

`docs/design/_handoff_to_implementation/tests/` に **Red テスト**があります。

- これらをプロジェクトのテストディレクトリに取り込み、**全テストが Red(失敗)** であることを確認してから実装に入る。
- テストを **Green にする最小実装**を書く。**テストを後から都合よく書き換えない**(テスト自体の誤り・仕様変更は設計 repo に戻って D-15 / TS-1 を改訂し、ゲートを再通過させてから反映)。

# {{案件名}} - 論理データモデル(論理ER図)

## 1. 目的
業務データを **正規化された形(通常は第3正規形)** で定義する。
本書は実装(DB 製品)に依存せず、業務的意味の単位でエンティティ・属性・関係を表現する。
DDD 採用時は DM-1 ドメインモデルの集約境界と整合させる(同名・同IDのエンティティで一致)。

## 2. 適用範囲・スコープ
- 含むもの: {{業務に登場する全エンティティ / 関係(リレーション)}}
- 含まないもの: {{物理化判断(非正規化・パーティション 等)→ D-8 物理データモデルで扱う}}

## 3. 前提・依存
- 参照ドキュメント: R-4 概念データモデル(あれば), R-13 用語集, DM-1 ドメインモデル(DDD採用時)
- 採番済ID: エンティティ E-* を本書で採番(R-4 で採番済なら流用)

### 4.3 属性定義
1エンティティにつき1ブロックで記述する。型は **論理型**(文字列 / 整数 / 日付 等)とし、物理型(VARCHAR / NUMBER 等)は D-9 テーブル定義書で確定する。

#### E-001: 顧客
| 属性 | 論理型 | NULL | 主キー / 外部キー | 制約 / 説明 |
|---|---|---|---|---|
| 顧客ID | 文字列(10) | NO | PK | システム採番 |
| 氏名 | 文字列(50) | NO | — | — |
| メール | 文字列(100) | NO | — | RFC5322 |
| 登録日 | 日付 | NO | — | — |

#### E-002: 受注
| 属性 | 論理型 | NULL | 主キー / 外部キー | 制約 / 説明 |
|---|---|---|---|---|
| 受注ID | 文字列(20) | NO | PK | システム採番 |
| 顧客ID | 文字列(10) | NO | FK → E-001 | — |
| 受注日 | 日付 | NO | — | — |
| 合計金額 | 整数 | NO | — | 0 以上 |

### 4.4 リレーション
| ID | 関係 | カーディナリティ | 必須性 | 業務的意味 |
|---|---|---|---|---|
| 顧客 — 受注 | 1 : N | 受注側必須 | 1顧客は0件以上の受注を持つ。1受注は必ず1顧客に属する |

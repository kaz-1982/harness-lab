# テンプレート共通フォーマット規約

`harness/templates/` 配下の全ドキュメントテンプレートはこの規約に従う。

---

## ファイル名規約

`{doc_id}_{ドキュメント名}.md`

例:
- `R-1_業務要件定義書.md`
- `B-12_論理データモデル.md`
- `D-15_単体テスト仕様書.md`
- `TS-1_受け入れテスト仕様書.md`(原典外・ハーネス拡張。Phase 3.5 / TDD採用時)
- `PR-1_事前検討サマリ.md`(原典外・ハーネス拡張。Phase 0 / 事前検討資料投入時)

ドキュメント名は `input/システム開発設計ドキュメント一覧.md` の表記に合わせる。記号(/、・等)は使用してよいが、ファイルシステムで問題になる場合は `_` に置換する。

---

## ファイル構造(標準テンプレート)

各テンプレートは以下の構造を **必ず** 含む。

~~~markdown
---
doc_id: {{ID 例: R-1}}
doc_name: {{名称}}
phase: {{事前検討 | 企画 | 要件定義 | 基本設計 | 詳細設計 | テスト | 横断}}
required_when: {{条件記述。例: "全規模・全手法で必須" / "UI有時必須"}}
depends_on: [{{前提ドキュメントID list 例: R-1, R-13}}]
mode: {{WF | Agile | 両対応}}
---

# {{案件名}} - {{ドキュメント名}}

## 1. 目的
このドキュメントの目的を1〜3行で。

## 2. 適用範囲・スコープ
- 含むもの:
- 含まないもの:

## 3. 前提・依存
- 参照ドキュメント: {{depends_on のID と名称}}
- 採番済ID: {{利用する機能ID、エンティティID等を列挙}}

## 4. 本文
(各ドキュメント固有のセクション。input資料の「記載内容」リストを見出し化する)

## 5. 関連トレーサビリティ
- 上流(本書の根拠となる要求・要件):
- 下流(本書を参照する設計・テスト):

## 6. 改訂履歴
| 版 | 日付 | 変更者 | 変更内容 |
|---|---|---|---|
| 0.1 | YYYY-MM-DD | | 初版作成 |

## 7. レビュー状態
- 単体品質チェック: 未 / 実施済(YYYY-MM-DD)
- 整合性チェック: 未 / 実施済(YYYY-MM-DD)
- TBD / 残課題:
  - (なし、または箇条書き)
- 承認: 未 / 承認(承認者、日付)
~~~

## 各テンプレ生成手順(Claude 向け)

新しいテンプレを作る必要が生じたら、以下の手順で `input/システム開発設計ドキュメント一覧.md` から派生させる。

### Step 1: 該当節を読む
input資料から該当ドキュメント(P-X, R-X, B-X, D-X)の節を読む。
含まれる項目は通常、**目的 / 記載内容 / 具体例**。

### Step 2: frontmatter を埋める
- `doc_id`: 該当ID
- `doc_name`: 該当ドキュメント名
- `phase`: input資料の所属フェーズ
- `required_when`: input資料の「必要度の目安」と `harness/01_selection_rules.md` を照合して条件を記述
- `depends_on`: 依存するドキュメントのID(下記参照)
- `mode`: 「両対応」を基本。アジャイルで省略可なら「WF」、アジャイル特有なら「Agile」

### Step 3: depends_on の判定
input資料のフェーズ順 + ドキュメント間の論理的な参照関係から判定。代表例:

| ドキュメント | depends_on の典型例 |
|---|---|
| R-2 As-Is/To-Be業務フロー図 | R-1 |
| R-4 概念データモデル | R-1 |
| DM-1 ドメインモデル(原典外) | R-4, R-13(R-4省略時 R-1, R-13) |
| R-5 ユースケース図 | R-1, R-3 |
| R-6 ユースケース記述 | R-5 |
| R-8 機能要件定義書 | R-1, R-5, R-6 |
| R-10 外部IF要件 | R-8 |
| B-1 システム方式設計書 | R-8, R-9, R-12 |
| B-7 画面一覧・遷移図 | R-8 |
| B-8 画面設計書 | B-7 |
| B-12 論理ER図 | R-4 |
| B-14 CRUD図 | B-6, B-12 |
| D-2 クラス図 | B-2 |
| D-4 シーケンス図 | R-6, D-2 |
| D-8 物理ER図 | B-12 |
| D-9 テーブル定義書 | D-8 |
| D-10 DDL | D-9 |
| D-12 モジュール仕様書 | D-2, D-4 |
| D-13 API詳細仕様書 | B-11 |
| D-15 単体テスト仕様書 | D-12, B-8(UI/コントローラ層) |
| TS-1 受け入れテスト仕様書(原典外) | R-7 or R-8(の AC), B-8 |
| R-13 用語集 | (なし、横断) |
| R-14 RTM | (横断、随時更新) |

不明な場合はユーザーに確認。

### Step 4: 「4. 本文」のサブセクション展開
input資料の「記載内容」リストの各項目を、`### 4.X {{項目名}}` の形でセクション化する。

例: R-1 業務要件定義書なら以下のサブセクションを生成:

~~~markdown
## 4. 本文

### 4.1 業務の目的・背景
(記入欄)

### 4.2 対象業務の範囲
(記入欄)

### 4.3 業務プロセスの概要
(記入欄)

### 4.4 業務ルール
(記入欄)

### 4.5 業務上のKPI・成功基準
(記入欄)

### 4.6 ステークホルダー
(R-15 ステークホルダー一覧 を参照、または列挙)
~~~

### Step 5: 「### 記入例」を末尾に追加
input資料の「具体例」を、本文の最後に補助的に転記:

~~~markdown
---

### 補足: 記入例(参考)

> {{input 資料の「具体例」をそのまま引用}}
~~~

### Step 6: テンプレ一覧を更新
`harness/templates/_index.md` の該当行に「生成済」マークを付ける。

## 原典外ドキュメントの例外規定(原典に節が無いドキュメント)

`TS-1 受け入れテスト仕様書` / `PR-1 事前検討サマリ` / `DM-1 ドメインモデル` / `ADR-NNNN アーキテクチャ決定記録` のような **ハーネス拡張(原典外)ドキュメント**は、`input/システム開発設計ドキュメント一覧.md` に該当節が無いため、上記 Step 1「該当節を読む」が成立しない。
この場合は典拠と本文構成を以下に読み替える。

> **構造の扱い**: TS-1 / PR-1 / DM-1 は frontmatter + セクション 1〜7 の標準構造を守る(check.py の標準検証対象)。
> **ADR のみ標準構造に従わない**(frontmatter は `adr_id`、本文は MADR 形式、check.py の標準検証対象外)。下記「ADR-NNNN」を参照。

### TS-1 受け入れテスト仕様書(Phase 3.5 / TDD採用時)
- **Step 1 の代替**: 典拠を「R-7 / R-8 の受け入れ基準(AC)」+「原典 §1.1 のV字テストレベル(統合テスト / 適格性確認テスト)」とする。
- **frontmatter**: `doc_id: TS-1`、`phase: テスト`、`required_when: "テスト戦略=TDD かつ AC定義あり"`、`depends_on: [R-7 or R-8, B-8]`。本文冒頭に「原典外・ハーネス拡張」と明記する。
- **本文**: 受け入れ基準を Given-When-Then 形式で 1ケース1行に展開し、各ケースに `AT-*` を採番する。
- テストコードそのものはテンプレ化せず、`_test_code_convention.md` の規約に従って `output/_handoff_to_implementation/tests/` に生成する。

### PR-1 事前検討サマリ(Phase 0 / 事前検討資料投入時)
- **Step 1 の代替**: 典拠を `input/事前検討資料/` に投入された資料(問題・原因・対応策を検討した資料)とする。原典資料の節ではなく、ユーザー提供の生資料を読み込む。
- **frontmatter**: `doc_id: PR-1`、`phase: 事前検討`、`required_when: "input/事前検討資料/ に資料が投入されている"`、`depends_on: []`(最上流)。本文冒頭に「原典外・ハーネス拡張」と明記する。
- **本文(4. 本文)**: `4.1 問題` / `4.2 原因` / `4.3 対応策` / `4.4 想定要件・制約・前提(要件候補のシード)` / `4.5 未確定事項(TBD)` で構成する。**資料に書かれていないことを補完しない**(推測が必要なら 4.5 に TBD として残す=ルール7)。
- 想定要件はあくまで **候補**。要件の確定は Phase 1 で行い、確定後に「5. 関連トレーサビリティ」の下流リンク(対応策 → R-*)を更新する。

### DM-1 ドメインモデル(Phase 1 / OO・DDD採用時)
- **Step 1 の代替**: 典拠を「DDD(ドメイン駆動設計)の戦術的設計パターン」+「R-4 概念データモデル」+「R-13 用語集(ユビキタス言語)」とする。
- **frontmatter**: `doc_id: DM-1`、`phase: 横断`、`required_when: "OO/DDD採用時必須、それ以外はエンティティ数が多い時に推奨"`、`depends_on: [R-4, R-13]`(R-4 を省略する案件では `[R-1, R-13]`)。本文冒頭に「原典外・ハーネス拡張」と明記する。
- **本文(4. 本文)**: ユビキタス言語 / 境界づけられたコンテキスト(コンテキストマップ) / 集約・集約ルート / エンティティ / 値オブジェクト / ドメインサービス / ドメインイベント / リポジトリ / 不変条件・ビジネスルール / ドメインモデル図 で構成する。
- **ID 採番**: 集約に `AG-*`、値オブジェクトに `VO-*` を採番し `_id_registry.md` に登録する。エンティティは原典の `E-*`(R-4 / B-12 / B-13)を流用し、B-12 / B-13 と同一の `E-` を指すよう整合させる(二重定義しない)。
- 図は `diagrams/DM-1_domain_model.drawio.svg` として drawio.svg 形式で作成し、本文から参照する。

### ADR-NNNN アーキテクチャ決定記録(随時 / 意思決定のたび)
- **Step 1 の代替**: 典拠を「B-1 システム方式設計 / B-2 アーキテクチャ設計 等で行う設計判断」とする。1つの意思決定につき1ファイルを `output/横断/ADR/` に蓄積する。
- **構造の例外(重要)**: ADR は標準セクション 1〜7 に従わず、業界標準の MADR / Nygard 形式を用いる。frontmatter のキーも `doc_id` ではなく **`adr_id`** とする(check.py の状態突合を誤発火させないため)。`output/横断/ADR/` 配下は check.py の標準検証対象外。
- **frontmatter**: `adr_id: ADR-0001`、`title`、`status`(Proposed / Accepted / Deprecated / Superseded by ADR-XXXX)、`date`、`deciders`、`related`。
- **本文**: ステータス / コンテキスト(背景・課題) / 決定 / 検討した選択肢 / 影響(Consequences) / 関連。本文冒頭に「原典外・ハーネス拡張」と明記する。
- **索引**: `output/横断/ADR/_index.md` に「ADR-ID / タイトル / 状態 / 関連 / 決定日」を1行ずつ記録する。採番は4桁ゼロパディング(ADR-0001…)を基本とする。

## 記述スタイル・表記ルール

### 1文1行(改行規約)

本文の地の文は1文ごとに改行する(句点「。」のたびに改行する)。
1行が長くなると差分レビューがしにくく、読みづらさの原因になるため。
Markdown では文末の改行(ソフトブレイク)はレンダリング時に半角空白として扱われ、段落内では1文に連結される。
よってこの規約は **ソースの可読性・差分の見やすさのため** であり、表示結果は変わらない。
1文が極端に長い場合は、読点「、」や接続で意味の切れ目を作り、必要なら複数の文に分割する。

改行対象外(これらは従来どおり):

- 表のセル(セル内では改行しない)
- コードブロック(``` で囲った範囲)
- 見出し
- frontmatter(`---` で囲った範囲)
- 箇条書きの各項目(1項目=1行)
- 引用ブロック(`>` で始まる行)

### 図(drawio.svg を正本)

図(ER図、業務フロー、状態遷移、ドメインモデル、シーケンス等)は **`.drawio.svg` 形式を正本**とする。
配置は、各成果物と同じ階層の `diagrams/` サブフォルダに `{doc_id}_{図名}.drawio.svg` で置き、本文から `![説明](diagrams/{doc_id}_{図名}.drawio.svg)` で参照する。
`.drawio.svg` は SVG 画像なので Markdown プレビュー・GitHub・VS Code のどこでも表示でき、かつ Draw.io デスクトップ / VS Code 拡張「Draw.io Integration」で再編集できる(表示と編集を両立する)。
Mermaid は可読補助として本文に併記してよいが、**正本は drawio.svg** とする(レンダラー非対応の環境で表示されない問題を避けるため)。
ASCII art は軽微な補助図に限る。

### その他の表記

- 日付は ISO 8601(YYYY-MM-DD)
- ID は半角英数 + ハイフン
- 桁数指定は半角(全角の `100` ではなく `100`)
- 表(マークダウン)を多用してよい(レビューしやすい)

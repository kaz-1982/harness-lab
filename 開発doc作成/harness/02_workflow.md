## 0.5 事前検討の取り込み(Phase 0 / 任意)

要件定義の前段で「問題・原因・対応策」を検討した資料が `input/事前検討資料/` に投入されている場合のみ実施する。**資料が無い案件はこの節をスキップ**し、§1 インテークへ直接進む。全体像は [WORKFLOW.md](../WORKFLOW.md) の Phase 0 を参照。

### 入口判定(許可拡張子 allowlist 方式 / v0.9〜)

1. `input/事前検討資料/` を確認する。
2. 以下の **許可拡張子に該当するファイルが1つ以上**ある場合のみ Phase 0 を実施:
   - `.md`, `.txt`(テキスト / Markdown)
   - `.docx`, `.pptx`, `.xlsx`(Office 文書)
   - `.pdf`(PDF)
   - `.png`, `.jpg`, `.jpeg`, `.gif`, `.svg`(画像)
3. 以下は **トリガ対象外**(OS / エディタ生成ファイルの誤発火を防ぐため):
   - 隠しファイル: `.DS_Store`, `Thumbs.db`, `desktop.ini`
   - エディタ一時ファイル: `~$xxx.docx`, `*.tmp`, `*.swp`, `*.bak`
   - プレースホルダ: `.gitkeep`, `.keep`
   - 説明ファイル: `README.md`
4. 許可拡張子に該当するファイルが1件も無ければスキップ(`project_profile.md` の「0. 事前検討資料」に「なし」と記録)。
5. allowlist に無い形式の資料(`.key`, `.numbers`, 音声・動画 等)がある場合は、ユーザーに「テキスト化 or PDF 化して再投入」を依頼するか、要点を口頭で補ってもらう(推測しない)。

### Step 0-A: 資料の読み込み

- `input/事前検討資料/` 配下の全ファイルを読む。
- md / テキスト → そのまま読む。docx / pptx / xlsx / pdf / 画像 → 対応するスキル(`docx` / `pptx` / `xlsx` / `pdf`)や画像読み込みで内容を抽出する。
- 形式が読めない資料があれば、ユーザーに要点を口頭で補ってもらう(推測しない)。

### Step 0-B: PR-1 事前検討サマリの生成

1. テンプレ `harness/templates/PR-1_事前検討サマリ.md` を Read(未確認なら必ず先に読む。テンプレ準拠=ルール5)。
2. 資料の内容を以下に正規化して `output/00_事前検討/PR-1_事前検討サマリ.md` を生成:
   - **4.1 問題(課題)**: 現状で何が問題か(事実ベース)。
   - **4.2 原因**: その問題がなぜ起きているか(根本原因の分析)。
   - **4.3 対応策**: 問題に対して検討された打ち手・方針。
   - **4.4 想定要件・制約・前提**: 対応策から導かれる、システムへの要求・制約・前提条件の候補(R-* のシード)。あくまで「候補」であり、要件としての確定は Phase 1 で行う。
   - **4.5 未確定事項 / インテークで確認すること**: 資料だけでは決められない点を TBD として列挙。
3. **資料に書かれていないことを補完しない。** 推測が必要な箇所は 4.5 に TBD として残す(ルール7)。
4. 「5. 関連トレーサビリティ」に「対応策 → 想定要件候補」の対応を記録(後で R-* が確定したら下流リンクを更新)。
5. Step F 相当の整合性チェック(`python3 harness/tools/check.py` + `harness/03_quality_checklist.md` の手動観点)を実施し、`output/_review_log.md` に記録。

### Step 0-C: ユーザーとの認識合わせ

- 生成した PR-1 の要約(問題・原因・対応策・想定要件候補)をユーザーに提示。
- 「この理解で合っていますか / 補足・訂正はありますか」と確認し、フィードバックを PR-1 に反映。

### Step 0-D: profile への記録 と インテークへの接続

- `project_profile.md` の「0. 事前検討資料」に、有無・参照ファイル名・PR-1 へのリンクを記録。
- 以降のインテーク(§1)では、PR-1 から読み取れる項目は空欄質問ではなく「資料からこう読み取りました。正しいですか?」の**確認形式**にして、ユーザーの回答負担を減らす。
- 要件定義(R-1 / R-7 / R-8 等)を作る際は、各ドキュメントの「5. 関連トレーサビリティ」上流に PR-1 を記載し、対応策 → 要件 の追跡性を残す。

完了基準:
- (資料あり)`output/00_事前検討/PR-1_事前検討サマリ.md` が生成され、機械チェック緑、ユーザーと認識が合っている
- (資料なし)`project_profile.md` に「事前検討資料: なし」が記録されている
- `_doc_plan.md` 生成時に PR-1 を1行追加(資料ありの場合)

## 3〜5. Phase 1 各フェーズ進行(共通手順)

各フェーズで `_doc_plan.md` 上の `担当: CC1` のドキュメントを順次作成。**1ドキュメントずつ** 以下の手順を踏む。

### Step A: テンプレ確認 / 生成
1. `harness/templates/{doc_id}_*.md` を Read
2. 存在しなければ:
   - `harness/templates/_format.md` の規約を読む
   - `input/システム開発設計ドキュメント一覧.md` の該当節(目的/記載内容/具体例)を読む
   - 規約に沿ったテンプレを生成
   - `harness/templates/_index.md` を更新

### Step B: 依存関係確認
1. テンプレ frontmatter の `depends_on` を確認
2. 前提ドキュメントが「作成済」または「承認済」であることを確認
3. 未完なら、ユーザーに「前提が未完了ですが進めますか?」と確認(原則: 待つ)

### Step C: 内容記入(対話)
1. テンプレのセクションごとに、必要情報をユーザーに質問
2. 1つの質問に答えが返ったら、該当セクションを更新
3. 業務固有の情報・要件・制約は推測せず必ず質問
4. 技術的常識(プロトコル仕様等)は Claude が記入してよい
5. **画面影響B-*の場合**: Phase 1 では「画面別の詳細は Phase 3 で確定」と明記し、方針レベルの記述に留める

### Step D: ID採番
1. 新規IDが発生したら `output/_id_registry.md` に追記
2. 採番ルール: `CLAUDE.md` の「3. ID体系の一貫性」を参照
3. 画面ID(S-*)は ClaudeDesign 側で採番されるため、Phase 1 では **採番しない**

### Step E: 用語登録
1. ドキュメント内に登場した業務用語を `output/横断/R-13_用語集.md` と突合
2. 未登録の用語は追加。表記揺れがあればユーザーに統一を確認

### Step F: 整合性チェック
1. **機械チェック** `python3 harness/tools/check.py` を実行(構造・ID整合・状態突合を自動検証)。検出された問題を解消する(緑になるまで)
2. `harness/03_quality_checklist.md` の **手動観点**(機械では測れない意味の妥当性など)を確認
3. 検出された問題はその場で修正、または「7. レビュー状態」に記録
4. 結果(機械チェック出力 + 手動確認)を `output/_review_log.md` に追記

### Step G: 完了マーク
1. `_doc_plan.md` の該当行の `状態` を `作成済` に更新
2. ユーザーレビューが必要な場合は `レビュー中` とし、レビュー後に `承認済`

### DM-1 ドメインモデル(OO/DDD採用時)
- R-4 概念データモデルの後(R-4 省略案件は R-1 の後)に、Step A〜G の通常手順で作成する。
- 集約 `AG-*` / 値オブジェクト `VO-*` を採番し `_id_registry.md` に登録する。エンティティは原典の `E-*`(R-4 / B-12 / B-13)を流用する。
- ドメインモデル図は drawio.svg 形式で `diagrams/` に作成し、本文から参照する。

### ADR アーキテクチャ決定記録(随時 / Phase 1・3 共通)
通常の Step A〜G とは別の軽量フローで運用する。
アーキテクチャ・技術選定の意思決定が発生するたびに、以下を行う:

1. `output/横断/ADR/ADR-{連番4桁}_{タイトル}.md` を `harness/templates/ADR-NNNN_*.md` から作成する。
2. ステータス(Proposed / Accepted 等)・コンテキスト・決定・検討した選択肢・影響を記入する。
3. `output/横断/ADR/_index.md`(索引)に1行追記する。
4. 関連する B-2 等から ADR を相互参照する。

- ADR は `_doc_plan.md` 上では1行集約(状態 `蓄積中`)とし、個別 ADR は計画行に起こさない。
- ADR は MADR 形式で `check.py` の標準検証対象外(frontmatter は `adr_id`)。

## 9.5 Phase 3.5: テスト具体化 + レビューゲート(テスト戦略 = TDD の案件のみ)

`output/project_profile.md` の「テスト戦略」が **TDD** の案件でのみ実施する。Phase 3 完了後、Phase 4(実装)に渡す前に、設計ドキュメントからテストを先に作り上げ、2段階のレビューゲートでユーザー承認を得る。

> 入口ガード4項目・テスト依存図・ゲート完了基準は [WORKFLOW.md §7.5](../WORKFLOW.md) に集約。本書は手順のみ。

入口ガード4項目を1つでも満たさない場合は着手せず、ユーザーに確認(技術スタック / テストフレームワークが TBD の場合は `project_profile.md` を更新してから着手 — ルール7「推測で埋めない」)。

### 9.5-① テスト仕様の確定 → ★ゲート1

1. **D-15 単体テスト仕様書を最終化**: 全対象モジュール × テスト観点(同値分割・境界値・デシジョンテーブル・状態遷移)× ケース(入力 / 前提 / 期待結果)。各ケースに `UT-*` を採番し `_id_registry.md` に登録。
2. **TS-1 受け入れテスト仕様書を作成**: AC(R-7 / R-8)を入力に Given-When-Then 形式でケース化。各ケースに `AT-*` を採番し `_id_registry.md` に登録。テンプレは `harness/templates/TS-1_受け入れテスト仕様書.md`(原典外・`_format.md` の T 系列例外規定で生成)。
3. **R-14 RTM のテストID列を充填**(UT-*/AT-*)。RTM 省略時は D-15 / TS-1 の「5. 関連トレーサビリティ」で要件 ⇄ テストの対応を代替記録。
4. Step F の整合性チェック(`harness/03_quality_checklist.md` の「Phase 3.5 ゲート1」)を実行し `_review_log.md` に記録。
5. **★ゲート1**: テスト仕様をユーザーがレビュー・承認。`_doc_plan.md` の D-15 / TS-1 を `ゲート1承認` に。

### 9.5-② Red テストコード生成 → ★ゲート2

1. `UT-*`/`AT-*` に1対1対応する実行可能テストコードを生成(`project_profile.md` の技術スタック / テストフレームワーク準拠。規約は `harness/templates/_test_code_convention.md`)。
2. **全テストが Red(失敗)で実行できることを確認**(コンパイル/解釈は通り、アサーションで fail)。実行ログを `_review_log.md` に添付。
3. `output/_handoff_to_implementation/tests/{unit,acceptance}/` に配置し、`_test_manifest.md`(UT/AT ↔ テストファイル ↔ 要件 の対応表。雛形は `harness/templates/_test_manifest.md`)を作成。

## アジャイル運用時の差分

### スプリント開始時
1. 対象ストーリー(R-7)を確定
2. 必要に応じて R-9 非機能要件、B-2 アーキテクチャを更新
3. 画面が必要なストーリーがあれば、当該ストーリー単位で `_handoff_to_claude_design/sprint_NN/` を作って ClaudeDesign に渡す
4. 横断ドキュメント(API仕様、ER図、用語集)を更新

### スプリント中
- **通常**: 「ストーリー → 受け入れ基準 → 画面設計(ClaudeDesign) → 必要な詳細設計の追記 → 実装 → 単体テスト」をワンサイクル
- **TDD採用時**: 「ストーリー → 受け入れ基準(AC) → 画面設計(ClaudeDesign) → 詳細設計の追記 → **テスト仕様(D-15/TS-1)+ Red テストコード → ★統合ゲート(ユーザー承認)** → 実装で Green 化」をワンサイクル(Just Enough によりゲート1+2は1回に統合)
- 詳細設計は `output/sprint_NN/` に増分追記
- 横断ドキュメントは `output/横断/` を最新版で維持

### スプリント終了時
- そのスプリントで追加・変更した API、エンティティ、画面、用語を `output/横断/` に反映
- 整合性チェック実施

## ハイブリッド運用時の境界

- フェーズ 3〜5(企画 + 要件定義 + 基本設計画面以外): WF として完了
- フェーズ 6(ClaudeDesign 引き渡し): 一回 or スプリントごと
- フェーズ 7〜9(取り込み + 詳細設計): Agile としてスプリントで増分作成
- フェーズ 9.5(Phase 3.5 テスト具体化、TDD採用時): 設計以降の `mode` に従う(Agile なら統合ゲート / WF なら2段階厳格)

- 上流のドキュメントが未承認の状態で下流フェーズに入らない

## 緊急停止条件

以下の場合は作業を停止し、ユーザーに報告してください:

- ハーネス自身の構造的不備を発見した(質問票漏れ、選定ルール矛盾、テンプレ不整合等)
- ドキュメント間で重大な矛盾を発見した(例: R-1 と R-8 が矛盾)
- ClaudeDesign 成果物と要件定義の間に重大な乖離がある(画面で前提とする業務プロセスが R-1 と異なる等)
- ユーザー回答が複数の解釈を持ち、推測すると後段に重大な影響がある

報告内容:
1. 発見した問題
2. 影響範囲
3. 提案する解決方針(複数案)

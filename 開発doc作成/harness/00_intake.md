# インテーク質問票

このファイルの質問に順番に答えてもらい、`output/project_profile.md` を生成します。

Claude は以下の指針でインテークを実施してください:

- **質問1に入る前に、必ず Phase 0(事前検討資料の確認)を実施する**(下記「0. 事前検討資料の確認」)。
- **一度に全部聞かない**。1〜3個ずつ区切って質問する。
- ユーザーが「分からない」「未定」と答えた項目は **TBD** として記録し、保留扱いにする(後で確定可能)。
- 回答内容から推測して `output/_doc_plan.md` の選定に直接影響しそうな項目は、特に丁寧に確認する。
- このハーネスは **画面設計 (B-7, B-8) を ClaudeDesign に委譲** する3ツール協調フロー([../WORKFLOW.md](../WORKFLOW.md))で運用される前提。インテーク冒頭で必要に応じてユーザーに以下を案内:
  - 「画面設計は ClaudeDesign 側で行います。Phase 1 完了後に引き渡しパッケージを生成しますので、ClaudeDesign 用の環境(プロジェクト等)をご準備ください。」
- 項目8の **テスト戦略** で TDD を選んだ場合、Phase 3.5(テスト具体化 + レビューゲート)が発動する。この場合は技術スタックとテストフレームワーク(項目5)を Phase 3.5 着手までに確定する必要があるため、未定なら「後で確定が必要」と伝える。

## 0. 事前検討資料の確認(Phase 0 / 任意)

**質問1より前に必ず実施する。** 要件定義の前段で「問題・原因・対応策」を検討した資料が `input/事前検討資料/` に投入されているかを確認する。

1. `input/事前検討資料/` の中身を確認する(原典の `README.md` は説明ファイルなので対象外)。
2. **資料がある場合**:
   - すべて読み込む(md / テキストはそのまま、docx / pdf / 画像等は対応するスキル・ツールで読む)。
   - 内容を「問題 → 原因 → 対応策 → 想定要件・制約・前提」に正規化し、`harness/templates/PR-1_事前検討サマリ.md` をテンプレに `output/00_事前検討/PR-1_事前検討サマリ.md` を生成する(Phase 0 の手順詳細は [02_workflow.md](02_workflow.md) §0)。
   - 生成した PR-1 の要約をユーザーに提示し、「この理解で合っていますか / 補足はありますか」と認識を合わせる。
   - 以降の各質問では、**PR-1 から読み取れる項目を空欄で聞き直さず、「資料からはこう読み取りました。これで正しいですか」という確認形式**にして、ユーザーの負担を減らす。
   - `project_profile.md` の「0. 事前検討資料」セクションに、有無・参照ファイル・PR-1 へのリンクを記録する。
3. **資料が無い場合**:
   - Phase 0 はスキップ。`output/00_事前検討/` も PR-1 も作らない。
   - `project_profile.md` の「0. 事前検討資料」セクションに「なし」と記録し、通常どおり質問1から開始する。

> 事前検討資料は案件によって「ある / ない」のどちらもあり得る。**無いことは異常ではない**。資料が無い案件で推測の事前検討サマリを捏造してはならない(ルール7)。

## 1. 案件の基本情報

- 案件名 / プロジェクトコード
- 案件概要(1〜3行)
- 発注元(社内 / 社外 / 自社プロダクト)
- 予定スケジュール(着手予定日 / 本番予定日)

## 2. 規模

- 開発期間の目安
- 開発体制人数の目安
- → 規模区分を判定:
  - **小規模**: 〜6ヶ月、5人以下
  - **中規模**: 6ヶ月〜2年、10〜30人
  - **大規模**: 2年以上、50人以上

## 3. 開発手法

- ウォーターフォール / アジャイル(Scrum等) / ハイブリッド
- ハイブリッドの場合: どのフェーズが何を採用するか(例: 上流WF、開発以降Scrum)
- アジャイルの場合: スプリント長(週単位)

## 4. システム特性(YES / NO で回答)

| 項目 | 説明 |
|---|---|
| UI(画面)を持つか | エンドユーザー向け画面の有無 |
| 帳票出力があるか | PDF・Excel・紙など定型帳票 |
| バッチ処理があるか | 定期実行・夜間処理 |
| 他システム連携があるか | REST/SOAP/ファイル/MQ等 |
| データ移行があるか | 既存システムからの移行 |
| 状態管理が複雑か | ワークフロー、注文ステータス等の遷移を伴う業務 |
| アルゴリズム的に複雑な処理があるか | 計算ロジック、最適化、機械学習等 |
| 業務改革(BPR)を伴うか | As-Is/To-Be の業務プロセス変更 |

## 5. アーキテクチャ

- 設計パラダイム: オブジェクト指向 / 構造化設計 / どちらでもない
- DDD(ドメイン駆動設計)を採用するか: YES / NO(YES の場合、Phase 1 で DM-1 ドメインモデルを必須作成し、集約・値オブジェクト等を定義する)
- 想定アーキテクチャスタイル(任意): レイヤード / ヘキサゴナル / マイクロサービス / モノリス / その他
- 想定技術スタック(任意、決まっていれば): 言語、フレームワーク、DB、クラウド等
- テストフレームワーク(**テスト戦略で TDD を選ぶ場合は必須**): 例 JUnit5+Mockito / Vitest / pytest / RSpec 等

## 6. 業界・規制

- 業界: 金融 / 医療 / 公共 / 製造 / 流通 / IT・SaaS / その他
- 適用される規制(複数選択可):
  - 個人情報保護法
  - 電子帳簿保存法
  - GDPR
  - HIPAA
  - PCI-DSS
  - 業界固有(例: FISC、医薬品GxP)
  - なし
- 監査要件の厳しさ: 高 / 中 / 低

## 7. 既存システムとの関係

- 関係: 既存システム刷新 / 新規構築 / 既存への機能追加
- 並行稼働期間の有無 / 期間
- 移行対象データ規模(おおよその件数)

## 8. ドキュメント運用ポリシー

- 発注者レビューの対象範囲: 全ドキュメント / 上流(要件定義まで)のみ / 一部 / なし
- 契約形態: 請負 / 準委任 / 内製 / 不明
- 監査・契約上の長期保管要否: あり / なし
- ドキュメントの納品形式の指定があるか: あり(形式) / なし
- **テスト戦略**: TDD(テスト先行 + レビューゲート) / 通常(設計後に実装、テストは実装と並行) / 未定
  - **TDD を選ぶと Phase 3.5(テスト具体化 + 2段階レビューゲート)が発動**し、設計ドキュメントから先にテスト仕様 → Red テストコードを作りユーザー承認を得てから実装に渡す(詳細は [../WORKFLOW.md §7.5](../WORKFLOW.md))
  - TDD を選ぶ場合、技術スタックとテストフレームワーク(項目5)を Phase 3.5 着手までに確定する必要がある

## 9. 重視する品質特性(ISO/IEC 25010)

以下から重視するものを選択(複数可)。優先度1〜3を付ける:

1. 機能適合性(Functional Suitability)
2. 性能効率性(Performance Efficiency)
3. 互換性(Compatibility)
4. 使用性(Usability)
5. 信頼性(Reliability)
6. セキュリティ(Security)
7. 保守性(Maintainability)
8. 移植性(Portability)

## 10. 既知の制約条件

- 技術的制約(例: 「DB は Oracle 19c 固定」)
- 法的制約
- 予算上限
- 絶対納期(法改正対応など)
- 人員・組織制約

## 11. その他

## 完了後の処理

すべての回答を `output/project_profile.md` のテンプレートに反映してください。
反映後、続けて以下を実行:

1. `harness/01_selection_rules.md` を読み、プロファイルから作成対象ドキュメントを判定
2. 判定結果を `output/_doc_plan.md` として書き出し
3. ユーザーに「以下のドキュメントを作成する計画ですが、よろしいですか?」と提示
4. ユーザー承認後、`harness/02_workflow.md` の手順で着手

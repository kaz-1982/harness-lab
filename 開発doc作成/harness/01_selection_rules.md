# ドキュメント選定ルール

`output/project_profile.md` の値に基づき、作成すべきドキュメントを決定します。
Claude はこのルールを機械的に適用し、結果を `output/_doc_plan.md` に書き出してください。

判定方針:

- `input/システム開発設計ドキュメント一覧.md` の「必要度の目安」を踏襲
- **必須** → 作成する
- **該当時必須** → プロファイルの条件に該当すれば作成する
- **推奨** → 規模・手法・特性で判定(下表参照)

担当(`担当` 列)は3ツール協調フロー([WORKFLOW.md](../WORKFLOW.md))に基づき以下のいずれか:

- **CC0**: ClaudeCode (このハーネス) — Phase 0(事前検討)で作成(事前検討資料が投入された案件のみ)
- **CC1**: ClaudeCode (このハーネス) — Phase 1 で作成
- **CC1-P3**: ClaudeCode (このハーネス) — Phase 3 で作成 / 確定
- **CC1-P35**: ClaudeCode (このハーネス) — Phase 3.5 で作成 / 確定(テスト戦略=TDD の案件のみ)
- **CD**: ClaudeDesign — Phase 2 で作成(B-7, B-8 のみ)

`画面影響` 列は、Phase 1 で方針レベル記述、Phase 3 で画面設計を反映して確定するドキュメントを示します(`◯` が画面影響あり)。

## 0. 事前検討(Phase 0 / 条件付き)

`input/事前検討資料/` に事前検討資料(原典 `README.md` 以外のファイル)が1つ以上投入されている案件でのみ、以下を Phase 0 で作成する。資料が無い案件では作成しない(`_doc_plan.md` の「省略するドキュメント」に「事前検討資料なしのため」と記録)。

| ID | ドキュメント | 担当 | 画面影響 | 発動条件 |
|---|---|---|---|---|
| PR-1 | 事前検討サマリ(原典外・ハーネス拡張) | CC0 | | `input/事前検討資料/` に資料が投入されている |

- **PR-1** は原典(`input/...一覧.md`)に存在しないハーネス拡張ドキュメント。「問題 → 原因 → 対応策 → 想定要件・制約・前提」を正規化し、インテーク・要件定義(R-1 / R-7 / R-8)の入力シードにする。生成手順は `harness/templates/_format.md` の「原典外ドキュメントの例外規定」に従う。
- PR-1 は要件を**確定**するものではない。あくまで要件候補のシードであり、確定は Phase 1 で行う。

## 7. テスト具体化(Phase 3.5)で作成 — テスト戦略 = TDD のときのみ

`output/project_profile.md` の「テスト戦略」が **TDD** の案件でのみ、以下を Phase 3.5 で作成する。TDD でない案件では作成しない(D-15 は通常どおり Phase 3 で完成させる)。

| ID | ドキュメント | 担当 | 画面影響 | 発動条件 |
|---|---|---|---|---|
| D-15 | 単体テスト仕様書(最終化) | CC1-P35 | ◯ | 常に(TDD時は Phase 3.5 でケースを最終化) |
| TS-1 | 受け入れテスト仕様書(原典外・ハーネス拡張) | CC1-P35 | ◯ | AC(R-7/R-8)が定義されている |
| — | Red テストコード(`_handoff_to_implementation/tests/`) | CC1-P35 | — | 技術スタック + テストフレームワークが確定済 |

- **入口ガード**: 技術スタック / テストフレームワークが `project_profile.md` で未確定(TBD)の場合、Red テストコード生成(3.5-②)には進まずユーザーに確認する(ルール7)。
- **TS-1** は原典(`input/...一覧.md`)に存在しないハーネス拡張ドキュメント。生成手順は `harness/templates/_format.md` の「T 系列テンプレ生成の例外規定」に従う。
- テストケースID(`UT-` 単体 / `AT-` 受け入れ)は `output/_id_registry.md` に登録する。

## 8. ドメインモデル・ADR(原典外・ハーネス拡張)

原典(`input/...一覧.md`)に無いハーネス拡張ドキュメント。生成手順は `harness/templates/_format.md` の「原典外ドキュメントの例外規定」に従う。

| ID | ドキュメント | 担当 | 画面影響 | 発動条件 |
|---|---|---|---|---|
| DM-1 | ドメインモデル(DDD) | CC1 | | 設計パラダイム = OO/DDD で必須。それ以外はエンティティ数が多い / ドメインが複雑なとき推奨 |
| ADR-NNNN | アーキテクチャ決定記録 | CC1 | | 全規模で推奨。アーキテクチャ・技術選定の意思決定が発生するたび随時1ファイル作成(`output/横断/ADR/`) |

- **DM-1** は Phase 1 で R-4 概念データモデルの後に作成する(R-4 省略時は R-1 を上流とする)。集約 `AG-*` / 値オブジェクト `VO-*` を採番し、エンティティは原典の `E-*` を流用する。標準セクション 1〜7 に準拠する(`check.py` の標準検証対象)。
- **ADR** は意思決定のたびに1決定1ファイルを `output/横断/ADR/` に蓄積し、索引 `output/横断/ADR/_index.md` で管理する。MADR 形式で frontmatter は `adr_id`(`check.py` の標準検証対象外)。`_doc_plan.md` には ADR を**1行集約**で載せ、状態は `蓄積中` とする(個別 ADR は行に起こさない)。

## 再判定のトリガー

- 新たな特性(外部連携追加など)が判明した

# 実装(Phase 4)引き渡しパッケージ

このディレクトリは、ハーネス (Phase 3.5) から **実装(Phase 4・別リポジトリの ClaudeCode)** への引き渡しパッケージです。**テスト戦略 = TDD の案件でのみ生成**されます。

全体フロー: [../../WORKFLOW.md](../../WORKFLOW.md) §7.5 / §8

---

## 目的

TDD で実装を進めるため、設計ドキュメントから先に作り上げた **Red テストコード**(失敗する実行可能テスト)を実装リポジトリに引き渡します。実装側はこのテストを **Green にする形で**アプリケーションコードを書きます。

ゲート1(テスト仕様)・ゲート2(Red コード)のユーザー承認を経たものだけがここに置かれます。

---

## 構成(Phase 3.5 ゲート2承認後に Claude が生成)

```
_handoff_to_implementation/
├── README.md           ← このファイル(雛形)
├── tests/
│   ├── unit/           ← D-15 由来の単体テスト(UT-* に対応)
│   └── acceptance/     ← TS-1 由来の受け入れテスト(AT-* に対応)
└── _test_manifest.md   ← UT/AT ↔ テストファイル ↔ 要件 の対応表
```

ハーネス起動時にはこの README のみがあります。Phase 3.5 完了後、Claude が `tests/` と `_test_manifest.md` を生成します。テストコードの生成規約は [../../harness/templates/_test_code_convention.md](../../harness/templates/_test_code_convention.md) を参照。

---

## 実装リポジトリ側でやること(Green 化)

1. `tests/` をプロジェクトのテストディレクトリに取り込む(技術スタックは `_test_manifest.md` 記載のものと一致させる)
2. テストを実行し、**全テストが Red(失敗)** であることを確認(出発点)
3. テストを **Green にする最小限の実装**を書く(テストを満たすことをゴールにする)
4. 全テスト Green を確認したら、リファクタリング(Red を出さない範囲で整理)

**重要**: テストを後から都合よく書き換えないこと。テスト自体に誤り・仕様変更があれば、ハーネスに戻って D-15 / TS-1 を改訂し、ゲートを再通過させてから反映する。

# 経理 SaaS — ソフトウェアアーキテクチャ設計書

## 1. 目的
本 SaaS の内部構造とアーキテクチャ原則を定義する。
本書は D-* 詳細設計の土台であり、R-9 非機能要件(性能 / 信頼性 / セキュリティ)の実現方式を示す。
主要な技術選定は ADR-0001 を参照。

## 2. 適用範囲・スコープ
- 含むもの: アーキテクチャスタイル / 技術スタック / 主要パターン / 開発規約
- 含まないもの: 個別クラス・モジュールの内部実装(D-2 / D-12 で扱う)/ ネットワーク詳細(本 sample では割愛)

## 3. 前提・依存
- 参照ドキュメント: R-9 非機能要件定義書
- 採番済ID: なし(本書は方針定義)
- 関連 ADR: ADR-0001(言語選定 = TypeScript)

## 4. 本文

### 4.1 アーキテクチャスタイル
レイヤードアーキテクチャ(Presentation / Application / Domain / Infrastructure)を採用する。
小規模アジャイル向けの軽量構成で、DDD は採用しないが Domain レイヤーをフレームワーク非依存に保つ。

### 4.2 アーキテクチャ図
本 golden sample では図ファイルを省略(本物の案件では `diagrams/B-2_architecture.drawio.svg` を作成して参照する)。

### 4.3 レイヤ / モジュール分割と依存ルール

| レイヤ | 責務 | 依存先(許可) | 依存禁止 |
|---|---|---|---|
| Presentation | HTTP リクエスト処理・JSON 変換 | Application | Domain, Infrastructure |
| Application | ユースケース調停・トランザクション境界 | Domain | Infrastructure(直接) |
| Domain | ビジネスロジック・不変条件 | — | 他全レイヤ |
| Infrastructure | DB / 外部 API / メール / バッチ | Domain(IF 実装) | Presentation |

### 4.4 技術スタックの詳細

| 種別 | 製品 / ライブラリ | バージョン | 用途 |
|---|---|---|---|
| 言語 | TypeScript | 5.x | 全レイヤ(根拠は ADR-0001) |
| Web フレームワーク | NestJS | 10 | HTTP / DI |
| ORM | Prisma | 5 | DB アクセス |
| バリデーション | Zod | 3 | 入力検証 |
| ロガー | Pino | 8 | 構造化ログ |
| テスト | Vitest + Supertest | — | 単体 / E2E |
| インフラ | AWS ECS Fargate + RDS PostgreSQL 16 | — | 実行基盤 |

### 4.5 主要パターン
- リポジトリパターン: Domain → Infrastructure の依存性逆転
- DI: NestJS DI コンテナ
- トランザクション境界: Application レイヤのユースケースメソッド単位
- エラーハンドリング: HTTP 例外を Presentation で集約、業務エラーは Application で型付き例外

### 4.6 非機能観点の実現方法

| 非機能特性(R-NF-*) | 実現方法 | 関連 ADR |
|---|---|---|
| R-NF-001(CSV 出力 3 秒以内) | 集計クエリにインデックス + ストリーミング応答 | — |
| R-NF-002(可用性 99.9%) | ECS Fargate を 2 AZ にレプリケート、RDS Multi-AZ | — |
| R-NF-003(暗号化保管) | RDS at rest 暗号化(AES-256)+ TLS 1.3 必須 | — |

### 4.7 開発規約
- Linter: ESLint(@typescript-eslint/recommended)+ Prettier
- ブランチ戦略: trunk-based(main + 短命 feature ブランチ)
- レビュー / CI: GitHub PR + GitHub Actions(`pnpm test` + `pnpm lint`)

### 4.8 採用しなかった選択肢
| 選択肢 | 不採用理由 | ADR |
|---|---|---|
| Go | チームの既存スキルが TypeScript で揃っていたため | ADR-0001 |
| マイクロサービス | 規模が小さく単一サービスで十分 | — |

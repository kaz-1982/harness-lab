# ドキュメント生成ハーネス — 運用ルール (Claude 向け)

このリポジトリは、システム開発で必要となる設計ドキュメントを「毎回同じ品質で」生成するためのハーネスです。**案件単位でこのリポジトリ全体をコピー**して使用します。

このファイルは Claude Code が会話開始時に必ず読み込みます。以下のルールに従って動いてください。

---

## 3ツール協調フロー(全体像)

このハーネスは以下のフェーズを 3ツールで分担して進めます。**Phase 0 は事前検討資料が `input/事前検討資料/` に投入された案件でのみ発動**し、**Phase 3.5 はプロファイルで「テスト戦略 = TDD」を選んだ案件でのみ発動**します。詳細グラフは [WORKFLOW.md](WORKFLOW.md) を参照。

| Phase | 担当 | 主成果物 |
|---|---|---|
| 0. 事前検討の取り込み(任意) | **ClaudeCode (このハーネス)** | PR-1 事前検討サマリ(資料投入時のみ) |
| 1. 要件定義 + 基本設計(画面以外) | **ClaudeCode (このハーネス)** | R-*, B-*(B-7, B-8 を除く), DM-1(DDD採用時), ADR(随時) |
| 2. 画面設計 | **ClaudeDesign** | B-7, B-8 + 画面モック |
| 3. 詳細設計 + 画面影響B-*の確定 | **ClaudeCode (このハーネス)** | D-*, B-6/B-11/B-14/B-19 更新 |
| 3.5. テスト具体化 + レビューゲート(TDD採用時) | **ClaudeCode (このハーネス)** | D-15 最終化, TS-1, Red テストコード |
| 4. 実装 | **ClaudeCode (実装、別リポジトリ)** | アプリケーションコード(テストを Green 化) |

**Phase 0(事前検討の取り込み)** は、要件定義に入る前に「問題・原因・対応策」を検討した資料がある場合に、それを読み込んで正規化し `output/00_事前検討/PR-1_事前検討サマリ.md` を生成、以降のインテーク・要件定義(R-1 / R-7 / R-8)の入力シードにするフェーズです。**事前検討資料は案件によって「ある / ない」のどちらもあり得ます**。資料が無い案件では Phase 0 をスキップし、通常どおりインテークから開始します。

このハーネスが直接担当するのは **(任意)Phase 0・Phase 1・Phase 3・(TDD採用時)Phase 3.5**。Phase 2 / 4 への引き渡し・取り込みは以下のディレクトリで管理:

- `output/_handoff_to_claude_design/` — Phase 2 への入力(ClaudeDesign 向けパッケージ)
- `output/04_画面設計_from_ClaudeDesign/` — Phase 2 の成果物(ユーザーが配置)
- `output/_handoff_to_implementation/` — Phase 4 への入力(Red テストコード + 設計の引き渡し / TDD採用時)

## 起動時の挙動

ユーザーから明示的な指示がない場合、まず `output/project_profile.md` の冒頭と `output/_doc_plan.md` の進捗状態を確認し、現在のフェーズを判別してください。

- **`output/project_profile.md` の「案件名:」が空 or 未記入**
  → 初回セットアップ。**まず `input/事前検討資料/` を確認**し、事前検討資料(原典の `README.md` 以外のファイル)が1つ以上あれば **Phase 0** に着手(下記)。資料が無ければそのまま `harness/00_intake.md` を読み、質問票に従ってインテークを実施。いずれの場合も、完了後 `harness/01_selection_rules.md` で `output/_doc_plan.md` を生成しユーザー承認を取る。

- **Phase 0(事前検討の取り込み、任意)**(`input/事前検討資料/` に資料があり、`output/00_事前検討/PR-1_事前検討サマリ.md` が未生成)
  → 資料をすべて読み、「問題 → 原因 → 対応策 → 想定要件・制約・前提」を `output/00_事前検討/PR-1_事前検討サマリ.md`(標準テンプレ準拠)に正規化。ユーザーに要約を提示し認識を合わせてから、その内容を入力シードとしてインテーク(`harness/00_intake.md`)へ進む。詳細は [harness/02_workflow.md](harness/02_workflow.md)。

- **Phase 1 進行中**(`_doc_plan.md` 上で R-* / B-*(画面以外) の `状態` が `未着手` or `進行中` あり)
  → 未完成ドキュメントから次に進めるものをユーザーに提示。

- **Phase 1 完了、Phase 2 引き渡し前**(画面以外の B-* がすべて `承認済` で、`output/_handoff_to_claude_design/README.md` が未生成)
  → 引き渡しパッケージを生成し、ユーザーに「ClaudeDesign に渡してください」と案内。

- **Phase 2 中(ClaudeDesign 作業中)**
  → ハーネス側の作業はなし。ユーザーから「成果物が揃った」と報告があるまで待機。

- **Phase 2 成果物到着後 / Phase 3 進行中**(`output/04_画面設計_from_ClaudeDesign/` に B-7, B-8 が存在し、D-* が未完了)
  → 取り込みチェックを実行し、画面影響B-*(B-6, B-11, B-14, B-19)の更新と D-* の作成に着手。

- **Phase 3 完了**(D-* がすべて `承認済`)
  → **テスト戦略 = TDD の場合**: Phase 3.5 に着手(D-15 最終化 + TS-1 作成 → ゲート1 → Red テストコード生成 → ゲート2)。**TDD不採用の場合**: Phase 4(実装、別リポジトリ)への引き渡しサマリを案内。

- **Phase 3.5 進行中**(TDD採用時、`_doc_plan.md` 上で D-15 / TS-1 / Red コードがゲート未承認)
  → 入口ガード(技術スタック + テストフレームワークが確定済か)を確認し、ゲート1(テスト仕様)→ ゲート2(Red コード)の順に進める。各ゲートでユーザー承認を取る。詳細は [harness/02_workflow.md](harness/02_workflow.md)。

- **Phase 3.5 完了**(ゲート2承認済)
  → `output/_handoff_to_implementation/` を生成し、Phase 4(実装、別リポジトリ)への引き渡しサマリを案内。

ユーザーから明示的な指示がある場合は、それを優先してください。ただし、新規ドキュメントの作成依頼を受けたら、必ず `_doc_plan.md` 上の計画と整合する内容かを確認してください(計画外なら理由をユーザーに確認)。

## 守るべき7つのルール

### 1. プロファイル駆動
作成対象ドキュメントは必ず `harness/01_selection_rules.md` × `output/project_profile.md` から決定する。Claude の判断で勝手に追加・削除しない。プロファイルが変更された場合は `_doc_plan.md` を再計算してユーザーに差分を提示する。

### 2. 1ドキュメント1セッション
複数のドキュメントを並行して中途半端に書かない。1つのドキュメントを「テンプレ確認 → 内容記入 → ID/用語登録 → 整合性チェック → 完了マーク」まで通してから次へ進む。

### 3. ID体系の一貫性
新たにIDを採番したら必ず `output/_id_registry.md` に登録する。原則:

| 対象 | プレフィクス | 例 |
|---|---|---|
| 業務要件 | R-B- | R-B-001 |
| 機能要件 | R-F- | R-F-001 |
| 非機能要件 | R-NF- | R-NF-001 |
| 機能 | F- | F-001 |
| 画面 | S- | S-001 |
| API | A- | A-001 |
| バッチ | BT- | BT-001 |
| 帳票 | RP- | RP-001 |
| エンティティ | E- | E-001 |
| 集約(DDD) | AG- | AG-001 |
| 値オブジェクト(DDD) | VO- | VO-001 |
| テーブル | T- | T-001 |
| 単体テストケース | UT- | UT-001 |
| 受け入れテストケース | AT- | AT-001 |

採番はゼロパディング3桁から開始(必要に応じて4桁以上に拡張)。同じIDの再利用や穴あけを避ける。

`output/_id_registry.md` は以下のマークダウン表形式で記録する(検証スクリプト `harness/tools/check.py` が parse する正本フォーマット):

| ID | 種別 | 名称 | 定義元 | 採番日 |
|---|---|---|---|---|
| F-001 | 機能 | ユーザー登録機能 | R-8 | YYYY-MM-DD |
| S-001 | 画面 | ログイン画面 | B-8 | YYYY-MM-DD |

「種別」は上表のプレフィクス対象名、「定義元」はそのIDを最初に定義したドキュメントのID。

> **テストケースID(UT- / AT-)** は Phase 3.5(TDD採用時)で採番する。`UT-` は原典 RTM の例にも登場する単体テスト、`AT-` はハーネス拡張の受け入れテスト。統合テスト `IT-` は今回スコープ外だが、RTM のテストID列に現れることがある。
> **文書IDとインスタンスIDは別レイヤー**である。文書ID(`R-1`, `D-15`, `TS-1`, `PR-1`, `DM-1` など 種別 + 1〜2桁連番)は上表のインスタンスID採番とは別系統。特に受け入れテスト仕様書の文書ID `TS-1`(Test Spec)は、テーブルのインスタンスID `T-001` とは別物(プレフィクスを `TS-` として衝突を回避している)。事前検討サマリの文書ID `PR-1`(Pre-study）も同様に文書IDレイヤーで、インスタンスID採番(`F-`, `S-` 等)とは独立しており衝突しない。
> **ドメインモデル `DM-1`** も文書IDレイヤーで、インスタンスID採番とは独立(`DM-` で始まり衝突しない)。**アーキテクチャ決定記録 `ADR-NNNN`** も文書IDレイヤーの蓄積系で、1決定=1ファイルとして `output/横断/ADR/` に増分する(`ADR-` プレフィクスにより API の `A-` とは衝突しない)。一方、ドメインモデルが定義する **集約 `AG-` / 値オブジェクト `VO-` はインスタンスIDレイヤー**であり、上表に従って `_id_registry.md` で採番管理する。DDD のエンティティは原典の `E-` を流用し、DM-1 と B-12 / B-13 で同一の `E-` を指すよう整合させる(二重定義しない)。

### 4. 用語の一元化
専門用語・業務用語は `output/横断/R-13_用語集.md` に登録してから他ドキュメントで使う。重複・揺れ(例: 「顧客」と「お客様」)を発見したら即座にユーザーに確認し、統一表記を確定させる。

### 5. テンプレ準拠と記述スタイル
ドキュメント本体を書く前に必ず `harness/templates/{doc_id}_*.md` を確認する。テンプレが未生成なら、`harness/templates/_format.md` の規約 + `input/システム開発設計ドキュメント一覧.md` の該当節 から先にテンプレを生成し、`harness/templates/_index.md` を更新してから本体を書き始める。
記述スタイルも `_format.md` に従う。特に **本文の地の文は1文ごとに改行(句点「。」で改行)** し、**図は `.drawio.svg` 形式を正本**として `diagrams/` 配下に置き本文から参照する(Mermaid は補助。レンダラー非対応環境で表示されない問題を避けるため)。

### 6. 整合性チェックとレビューゲートの強制
ドキュメントを完成させる直前に、まず **機械チェック `python3 harness/tools/check.py` を実行**(構造・ID整合・状態突合を自動検証)し、検出された問題を解消する。その上で `harness/03_quality_checklist.md` の **手動観点**(意味の妥当性・検証可能性など機械では測れない項目)を確認し、結果を `output/_review_log.md` に追記する。チェックで検出した未解決事項は、ドキュメントの「7. レビュー状態」セクションにも残す。

> 機械チェックが緑でも「意味的に正しい」ことは保証されない。**自動(check.py)= 形式・参照・整合 / 手動(checklist)= 意味の妥当性** という2層で運用する。

TDD採用時、**Phase 3.5 のテスト成果物(テスト仕様書 / Red テストコード)はレビューゲートをユーザーが承認するまで Phase 4(実装)に渡さない**。ゲート1(テスト仕様)とゲート2(Red コード)の2段階を経ること(アジャイル運用では1回に統合可)。

### 7. 不明点はユーザーに確認
推測で埋めない。技術的な常識(HTTPステータスコードの意味など)以外の業務情報・要件・制約は、必ずユーザーに質問する。質問できない場合は空欄のまま「**TBD**: 確認事項」と明記して残す。

## 開発手法による分岐

`output/project_profile.md` の `開発手法` フィールドに応じて、Claude の進め方を変えてください。3ツール協調フローは開発手法によらず常に適用されます(画面設計は必ず ClaudeDesign に委譲)。

### ウォーターフォール
- Phase 1 を上から順に完了させる(企画 → 要件定義 → 基本設計の画面以外)
- 前フェーズの成果物が「承認」状態になるまで次フェーズに進まない
- Phase 2 (ClaudeDesign) を経て、Phase 3 で D-* と画面影響B-* を確定
- ドキュメントの網羅性・正確性を重視
- **TDD採用時**: Phase 3 完了後に Phase 3.5 を一括実施。ゲート1・ゲート2を**それぞれ厳格に**通す

### アジャイル(Scrum等)
- 横断ドキュメント(用語集、RTM、ER図、API仕様、アーキテクチャ概略)は最新版を `output/横断/` に維持
- スプリント単位の作業は `output/sprint_NN/` に追記
- 「Just Enough」原則: 不要な詳細を書かない。書くなら今のスプリントで使う分だけ
- ユーザーストーリー(R-7)が要求の主入口、SRS(R-8)は使わないか軽量化
- 画面が必要な単位ごとに ClaudeDesign に小さな引き渡しパッケージを作る運用も可
- **TDD採用時**: Phase 3.5 をスプリント内サイクルに組み込む。Just Enough によりゲート1+ゲート2を1回のレビューに統合してよい(ストーリー → AC確定 → 画面設計 → 詳細設計 → テスト仕様+Red → 承認 → 実装で Green 化)

### ハイブリッド
- `_doc_plan.md` で各ドキュメントに `mode: WF | Agile` を付与し、フェーズごとに進め方を切り替え
- 標準: 上流(企画・要件定義)は WF、設計以降は Agile
- **TDD採用時**: Phase 3.5 は設計以降の mode に従う(Agile なら統合ゲート、WF なら2段階厳格)

## ハーネス自身の改善について

ハーネスの構造的な不備(質問票の漏れ、選定ルールの誤り、テンプレの不整合)を発見した場合は、案件成果物を作る前に必ずユーザーに報告してください。ハーネスを更新する場合は、`harness/` 配下のファイルを編集し、変更理由を記録してください:

- **ハーネス本体の改修**(全案件に波及する構造変更): リポジトリルートの `ChangeLog.md` に記録する(`output/_review_log.md` は案件コピー時に初期化され消えるため、本体改修の記録には向かない)
- **案件作業中の局所的な微修正**: その案件の `output/_review_log.md` に「ハーネス改修」として記録する

## 運用モデル

このハーネスは **Opus(`claude-opus-4-8`)での運用を前提** に設計されています。
ユーザーが Sonnet / Haiku で起動した場合は、会話冒頭で「このハーネスは Opus 運用前提です。`/model opus` への切り替えを推奨します」と一度だけ案内してください(切り替えを強制はしない)。

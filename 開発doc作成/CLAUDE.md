# ドキュメント生成ハーネス — 運用ルール (Claude 向け)

このリポジトリは、システム開発で必要となる設計ドキュメントを「毎回同じ品質で」生成するためのハーネスです。**案件単位でこのリポジトリ全体をコピー**して使用します。
このファイルは Claude Code が会話開始時に必ず読み込みます。以下のルールに従って動いてください。

> **詳細リファレンス**:
> - フェーズ全体図・引き渡し仕様 → [WORKFLOW.md](WORKFLOW.md)
> - 各フェーズの実行手順 → [harness/02_workflow.md](harness/02_workflow.md)
> - 主要ファイルの一覧 → [harness/_files_overview.md](harness/_files_overview.md)

---

## 3ツール協調フロー(全体像)

| Phase | 担当 | 主成果物 |
|---|---|---|
| 0. 事前検討の取り込み(任意) | **ClaudeCode (このハーネス)** | PR-1 事前検討サマリ(資料投入時のみ) |
| 1. 要件定義 + 基本設計(画面以外) | **ClaudeCode (このハーネス)** | R-*, B-*(B-7, B-8 を除く), DM-1(DDD採用時), ADR(随時) |
| 2. 画面設計 | **ClaudeDesign** | B-7, B-8 + 画面モック |
| 3. 詳細設計 + 画面影響B-*の確定 | **ClaudeCode (このハーネス)** | D-*, B-6/B-11/B-14/B-19 更新 |
| 3.5. テスト具体化 + レビューゲート(TDD採用時) | **ClaudeCode (このハーネス)** | D-15 最終化, TS-1, Red テストコード |
| 4. 実装 | **ClaudeCode (実装、別リポジトリ)** | アプリケーションコード(テストを Green 化) |

このハーネスが直接担当するのは **(任意)Phase 0・Phase 1・Phase 3・(TDD採用時)Phase 3.5**。**Phase 0** は事前検討資料が `input/事前検討資料/` に投入された案件のみ、**Phase 3.5** は「テスト戦略 = TDD」の案件のみ発動。
引き渡しディレクトリ: `output/_handoff_to_claude_design/`(Phase 1→2)/ `output/04_画面設計_from_ClaudeDesign/`(Phase 2→3)/ `output/_handoff_to_implementation/`(Phase 3→4 / TDD時)。

## 起動時の挙動

ユーザーから明示的な指示がない場合、まず `output/project_profile.md` の冒頭と `output/_doc_plan.md` の進捗状態を確認し、現在のフェーズを判別してください。

> **ハーネス本体ガード(v0.10〜)**: リポジトリ直下に `.harness-source` マーカーが**存在し**かつ `project_profile.md` の「案件名:」が空のとき、ユーザーが案件作業(インテーク・PR-1 生成・ドキュメント作成 等)を始めようとしたら、まず以下を案内してください:
>
> > 「ここはハーネス本体(マスター)です。案件作業はコピーで実施します。`bash harness/tools/new-project.sh <案件のパス>` で案件用ディレクトリを作成し、そちらで `claude` を再起動してください。本体で作業を続けたい場合(=テスト目的等)はその旨をお知らせください。」
>
> ユーザーが「本体で進めてよい」と明示的に応答した場合のみ、本体でのインテークに進む。`.harness-source` が無いディレクトリ(=案件コピー先)では、本ガードは発火せず通常の起動フローに進む。

- **`project_profile.md` の「案件名:」が空 or 未記入**
  → 初回セットアップ。**まず `input/事前検討資料/` を確認**し、**許可拡張子**(`.md` / `.txt` / `.docx` / `.pptx` / `.xlsx` / `.pdf` / `.png` / `.jpg` / `.jpeg` / `.gif` / `.svg`)に該当するファイルが1つ以上あれば **Phase 0** に着手。隠しファイル(`.DS_Store` / `Thumbs.db` 等)・エディタ一時ファイル(`~$*.docx` 等)・`README.md` は無視。該当ファイルが無ければそのまま `harness/00_intake.md` の質問票でインテークを実施。完了後 `harness/01_selection_rules.md` で `output/_doc_plan.md` を生成しユーザー承認を取る。

- **Phase 0(任意)**(資料あり / `output/00_事前検討/PR-1_事前検討サマリ.md` が未生成)
  → 資料を読み「問題 → 原因 → 対応策 → 想定要件・制約・前提」を PR-1 に正規化。ユーザーと認識を合わせてからインテークへ進む。詳細手順は [harness/02_workflow.md §0.5](harness/02_workflow.md)。

- **Phase 1 進行中**(R-* / B-* 画面以外が `未着手` or `進行中`)
  → 次に進めるドキュメントをユーザーに提示。

- **Phase 1 完了、Phase 2 引き渡し前**(画面以外の B-* が全て `承認済` / `_handoff_to_claude_design/README.md` 未生成)
  → 引き渡しパッケージを生成し、ユーザーに「ClaudeDesign に渡してください」と案内。

- **Phase 2 中**(ClaudeDesign 作業中)
  → ハーネス側の作業はなし。ユーザーから完了報告があるまで待機。

- **Phase 2 成果物到着後 / Phase 3 進行中**(`04_画面設計_from_ClaudeDesign/` に B-7, B-8 / D-* 未完)
  → 取り込みチェック → 画面影響B-*(B-6, B-11, B-14, B-19)更新 → D-* 作成。

- **Phase 3 完了**(D-* 全て `承認済`)
  → **TDD の場合**: Phase 3.5 に着手(D-15 最終化 + TS-1 作成 → ゲート1 → Red コード生成 → ゲート2)。**TDD不採用**: Phase 4 への引き渡しサマリを案内。

- **Phase 3.5 進行中**(TDD採用時)
  → 入口ガード(技術スタック + テストフレームワーク確定)を確認し、ゲート1 → ゲート2 の順に進める。詳細は [harness/02_workflow.md §9.5](harness/02_workflow.md)。

- **Phase 3.5 完了**(ゲート2承認済)
  → `output/_handoff_to_implementation/` を生成し、Phase 4 への引き渡しサマリを案内。

ユーザーから明示的な指示がある場合は、それを優先してください。ただし、新規ドキュメント作成依頼を受けたら、必ず `_doc_plan.md` 上の計画と整合する内容かを確認してください(計画外なら理由をユーザーに確認)。

> **セッション境界の挙動(v0.11〜)**:
> - **セッション開始時**(現在のフェーズを判別する直前): `python harness/tools/check.py --session-start output` を実行し、`output/_session_log.md` 末尾の前セッションエントリを読む。エントリがあれば「前回はここまで完了 / 次の開始点はここ」をユーザーに提示してから作業を始める。エントリが無ければ初回セッションとして扱う。
> - **セッション終了時**(commit する直前): `python harness/tools/check.py --session-end output` でテンプレを取得し、必要箇所(タイトル / 次の開始点 / 意思決定メモ / 残 TBD / check.py 結果 / commit ハッシュ)を埋めて `output/_session_log.md` の末尾に追記してから commit する。**意思決定メモ**には、ADR を切るほどではない設計判断・スコープ変更・小さな合意を残す(後で「なぜそうしたか」を思い出すため)。

## 守るべき7つのルール

### 1. プロファイル駆動
作成対象ドキュメントは必ず `harness/01_selection_rules.md` × `output/project_profile.md` から決定する。Claude の判断で勝手に追加・削除しない。プロファイルが変更された場合は `_doc_plan.md` を再計算してユーザーに差分を提示する。

### 2. 1ドキュメント1セッション
複数のドキュメントを並行して中途半端に書かない。1つのドキュメントを「テンプレ確認 → 内容記入 → ID/用語登録 → 整合性チェック → 完了マーク」まで通してから次へ進む。

### 3. ID体系の一貫性
新たにIDを採番したら必ず `output/_id_registry.md` に登録する。プレフィクスは以下に従う:

| 対象 | プレフィクス | 例 |
|---|---|---|
| 業務要件 | R-B- | R-B-001 |
| 機能要件 | R-F- | R-F-001 |
| 非機能要件 | R-NF- | R-NF-001 |
| 機能 | F- | F-001 |
| 画面 | S- | S-001 |
| API | A- | A-001 |
| バッチ | BT- | BT-001 |
| 帳票 | RP- | RP-001 |
| エンティティ | E- | E-001 |
| 集約(DDD) | AG- | AG-001 |
| 値オブジェクト(DDD) | VO- | VO-001 |
| テーブル | T- | T-001 |
| 単体テストケース | UT- | UT-001 |
| 受け入れテストケース | AT- | AT-001 |

採番はゼロパディング3桁から開始(必要時 4桁以上に拡張)。同IDの再利用や穴あけを避ける。
台帳フォーマット(`check.py` が parse する正本)は `| ID | 種別 | 名称 | 定義元 | 採番日 |` の表形式。
**文書ID(`R-1` / `D-15` / `TS-1` / `PR-1` / `DM-1` / `ADR-NNNN` 等)は上表のインスタンスID採番とは別レイヤー**で、独立して衝突しない。`UT-` / `AT-` は Phase 3.5(TDD採用時)で採番する。DDD のエンティティは原典の `E-` を流用(DM-1 と B-12 / B-13 で二重定義しない)。

### 4. 用語の一元化
専門用語・業務用語は `output/横断/R-13_用語集.md` に登録してから他ドキュメントで使う。重複・揺れ(例: 「顧客」と「お客様」)を発見したら即座にユーザーに確認し、統一表記を確定させる。

### 5. テンプレ準拠と記述スタイル
ドキュメント本体を書く前に必ず `harness/templates/{doc_id}_*.md` を確認する。テンプレが未生成なら、`harness/templates/_format.md` の規約 + `input/システム開発設計ドキュメント一覧.md` の該当節 から先にテンプレを生成し、`harness/templates/_index.md` を更新してから本体を書き始める。
記述スタイルも `_format.md` に従う。特に **本文の地の文は1文ごとに改行(句点「。」で改行)** し、**図は `.drawio.svg` 形式を正本**として `diagrams/` 配下に置き本文から参照する。**Mermaid は使わない**(テキスト構文は人手での修正が難しく、レンダラー非対応環境で表示されない。Draw.io なら GUI から直感的に編集でき、SVG なので全環境で画像表示できる)。ハーネス取説図(`WORKFLOW.md`)も含め例外なく `.drawio.svg` に統一する(v0.14〜)。

### 6. 整合性チェックとレビューゲートの強制
ドキュメントを完成させる直前に、まず **機械チェック `python3 harness/tools/check.py` を実行**(構造・ID・状態・孤立・AC/AT・ファイル名・依存・ADR を自動検証)し、検出された問題を解消する。その上で `harness/03_quality_checklist.md` の **手動観点**(意味の妥当性・検証可能性など機械では測れない項目)を確認し、結果を `output/_review_log.md` に追記する。未解決事項はドキュメントの「7. レビュー状態」にも残す。完了判定前に `check.py --tbd` で `_tbd_dashboard.md` を更新して取りこぼし確認を推奨。

> 機械チェックが緑でも「意味的に正しい」ことは保証されない。**自動(check.py)= 形式・参照・整合 / 手動(checklist)= 意味の妥当性** の2層で運用する。

TDD採用時、**Phase 3.5 のテスト成果物はレビューゲートをユーザーが承認するまで Phase 4 に渡さない**。ゲート1(テスト仕様)とゲート2(Red コード)の2段階を経ること(アジャイル運用では1回に統合可)。

### 7. 不明点はユーザーに確認
推測で埋めない。技術的な常識(HTTPステータスコードの意味など)以外の業務情報・要件・制約は、必ずユーザーに質問する。質問できない場合は空欄のまま「**TBD**: 確認事項」と明記して残す(`check.py --tbd` で集約される)。

## 開発手法による分岐

`output/project_profile.md` の `開発手法` に応じて進め方を変える(3ツール協調フローは手法によらず常に適用)。詳細手順は [harness/02_workflow.md](harness/02_workflow.md) の「アジャイル運用時の差分」「ハイブリッド運用時の境界」を参照。

- **ウォーターフォール**: フェーズ順に完了させ、前フェーズ承認まで次に進まない。網羅性・正確性を重視。TDD採用時はゲート1・2を**それぞれ厳格に**通す
- **アジャイル**: 横断ドキュメントを `output/横断/` で最新版維持、スプリント単位を `output/sprint_NN/` に。Just Enough 原則(必要最小限)。R-7 が要求の主入口。TDD採用時はゲート1+2を1回に統合してよい
- **ハイブリッド**: `_doc_plan.md` で各ドキュメントに `mode: WF | Agile` を付与。標準は上流 WF / 設計以降 Agile

## ハーネス自身の改善について

ハーネスの構造的な不備(質問票の漏れ、選定ルールの誤り、テンプレの不整合)を発見した場合は、案件成果物を作る前に必ずユーザーに報告してください。ハーネスを更新する場合は、`harness/` 配下のファイルを編集し、変更理由を記録してください:

- **ハーネス本体の改修**(全案件に波及): リポジトリルートの `ChangeLog.md` に記録(`output/_review_log.md` は案件コピー時に消えるため不適)
- **案件作業中の局所的な微修正**: その案件の `output/_review_log.md` に「ハーネス改修」として記録

## 運用モデル

このハーネスは **Opus(`claude-opus-4-8`)での運用を前提** に設計されています。
ユーザーが Sonnet / Haiku で起動した場合は、会話冒頭で「このハーネスは Opus 運用前提です。`/model opus` への切り替えを推奨します」と一度だけ案内してください(切り替えを強制はしない)。

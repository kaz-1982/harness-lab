# システム開発 設計ドキュメント完全ガイド

## 本資料について

本資料は、ソフトウェア工学・要求工学(Requirements Engineering)・IPA共通フレーム2013(SLCP-JCF)・ISO/IEC/IEEE 12207・UML・ISO/IEC 25010 などの学術的枠組みに基づき、システム開発の各フェーズで作成されるドキュメントを体系的に整理したものです。

実プロジェクトでは全てを作成する必要はなく、規模・開発手法(ウォーターフォール/アジャイル)・業界慣習に応じて取捨選択します。本資料は「全体像を把握した上で、必要なものを選ぶ」ための参照資料として使用することを想定しています。

## 1. 全体像とフェーズの位置づけ

### 1.1 共通フレーム2013における位置づけ

```
[企画プロセス]
  ├─ システム化構想の立案
  └─ システム化計画の立案
        ↓
[要件定義プロセス]
  ├─ 業務要件定義
  ├─ 機能要件定義
  └─ 非機能要件定義
        ↓
[開発プロセス]
  ├─ システム方式設計 ─┐
  ├─ ソフトウェア方式設計 ─┴─ 基本設計(外部設計)
  ├─ ソフトウェア詳細設計 ── 詳細設計(内部設計)
  ├─ 実装
  ├─ 統合テスト
  └─ 適格性確認テスト
```

### 1.2 設計の3つの視点(構造化分析・Yourdon法)

いずれのフェーズでも、以下の3視点で整理すると漏れが少なくなります。

| 視点 | 英語 | 代表的な成果物 |
|---|---|---|
| データ | What | ER図、データ辞書、CRUD図 |
| 機能 | How | DFD、機能階層図、ユースケース図 |
| 制御(振る舞い) | When/Control | 状態遷移図、アクティビティ図、シーケンス図 |

### 1.3 設計の2軸(UMLの分類)

UMLの13種類の図は以下の2軸で整理されています。

- **構造図(Structure Diagrams)**: クラス図、オブジェクト図、コンポーネント図、配置図、パッケージ図、コンポジット構造図、プロファイル図
- **振る舞い図(Behavior Diagrams)**: ユースケース図、アクティビティ図、状態遷移図、シーケンス図、コミュニケーション図、タイミング図、相互作用概要図

## 3. 企画プロセス(要件定義の前段)

### P-1. システム化構想書

**目的**  
経営・事業戦略の観点から、システム化すべきか否かの方針を決定する。なぜやるのかを定義する最上流のドキュメント。

**記載内容**  
- 経営課題・事業課題
- システム化の方向性
- 期待される効果(ROI、KPI)
- 投資規模の概算
- リスク

**具体例**  
「当社の受注業務は年率20%で増加しているが、現行の手作業運用では処理能力の限界が近い。今後3年以内に受注プロセスのデジタル化投資を行わなければ機会損失が年間5億円に達する見込み。よって基幹受注システムの刷新を構想する。」

---

### P-2. システム化計画書

**目的**  
構想を具体的な実行計画に落とし込む。いつ・誰が・何を・どう進めるかを定義。

**記載内容**  
- システム化の範囲
- スケジュール・マイルストーン
- 体制
- 予算
- 調達方針(自社開発/パッケージ/SaaS)
- リスク管理方針

**具体例**  
「2026年度上期:要件定義、下期:基本設計・開発、2027年度上期:テスト・移行、下期:本番稼働。総予算3億円。開発は外部ベンダー選定のうえRFPを発行。」

---

### P-3. プロジェクト計画書

**目的**  
PMBOK等のプロジェクトマネジメント標準に基づき、プロジェクト遂行のための計画を文書化。

**記載内容**  
- プロジェクト憲章(目的、成功基準)
- WBS(Work Breakdown Structure)
- スケジュール(ガントチャート)
- 体制図・役割定義
- コミュニケーション計画
- リスク管理計画
- 品質管理計画
- 変更管理プロセス

**具体例**  
WBSを第3階層まで分解し、各タスクに担当・期間・成果物を割り当てる。例:「1.要件定義 > 1.1 業務要件定義 > 1.1.1 As-Is業務ヒアリング(担当:A氏、期間:1週間、成果物:ヒアリング議事録)」。

## 4. 要件定義フェーズ

要件工学(Requirements Engineering)は、要求獲得(Elicitation)→要求分析(Analysis)→要求仕様化(Specification)→要求検証(Validation)→要求管理(Management)の5工程で構成されます。

### 4.1 業務要件(Business Requirements)レベル

#### R-1. 業務要件定義書

**目的**  
発注者(事業部門)の業務上の要求をシステム観点を交えずに定義する。開発者と発注者の最初の合意点。

**記載内容**  
- 業務の目的・背景
- 対象業務の範囲
- 業務プロセスの概要
- 業務ルール
- 業務上のKPI・成功基準
- ステークホルダー

**具体例**  
「経理部の月次決算業務において、現在5営業日を要している売掛金集計処理を、新システム導入により1営業日以内に完了させる。対象範囲は国内法人顧客の売掛金管理のみとし、海外法人・個人顧客は対象外とする。」

---

#### R-2. As-Is/To-Be業務フロー図

**目的**  
現状業務(As-Is)と将来業務(To-Be)を可視化し、変化点を明確にする。業務改革(BPR)を伴うプロジェクトで特に重要。

**記載内容**  
- BPMN記法またはスイムレーン図による業務プロセス
- 担当部門・担当者の明示
- 情報の流れ(伝票、データ)
- システム/手作業の区分

**具体例**  
As-Is:「顧客→営業(紙の注文書受領)→営業事務(Excel入力)→上長(押印承認)→経理(再入力)」という5ステップ・3日間の流れ。  
To-Be:「顧客→Webポータル直接入力→ワークフローで上長電子承認→経理システム自動連携」という3ステップ・即時処理。

---

#### R-3. 業務機能関連図

**目的**  
業務を機能単位に分解し、機能間の関係を階層的に整理する。

**記載内容**  
- 業務機能の階層構造
- 機能間の入出力関係

**具体例**  
「受注管理業務」を「受注登録」「受注変更」「受注取消」「受注照会」に分解し、それぞれの入出力データを整理。

---

#### R-4. 概念データモデル

**目的**  
業務で扱う主要な概念(エンティティ)と関係を、実装に依存しない高い抽象度で整理する。

**記載内容**  
- 主要エンティティ(名詞抽出)
- エンティティ間の関係
- 主要な属性(必要最小限)

**具体例**  
「顧客」「商品」「受注」「請求」というエンティティを抽出し、「顧客は複数の受注を持つ」「1つの受注は複数の商品を含む」「1つの受注は1つの請求を発生させる」という関係を図示。

---

### 4.2 利用者要件(User Requirements)レベル

#### R-5. ユースケース図(UML)

**目的**  
システムの利用者(アクター)とシステムが提供する機能(ユースケース)の関係を俯瞰する。

**記載内容**  
- アクター(人・外部システム)
- ユースケース(楕円)
- 関連(関連線、include、extend、汎化)

**具体例**  
アクター「顧客」「営業担当」「管理者」に対して、「商品を検索する」「注文する」「注文状況を確認する」「顧客マスタを保守する」などのユースケースを配置し、線で結ぶ。

---

#### R-6. ユースケース記述

**目的**  
ユースケース図の各ユースケースについて、処理の流れを自然言語で詳細化する。

**記載内容**  
- ユースケース名、概要
- アクター
- 事前条件、事後条件
- 主シナリオ(基本フロー)
- 代替シナリオ・例外シナリオ
- 頻度、優先度

**具体例**  
ユースケース「注文する」:  
- 事前条件:顧客がログイン済み  
- 主シナリオ:①顧客が商品をカートに追加 → ②「注文確定」ボタン押下 → ③システムが在庫確認 → ④システムが注文番号を採番 → ⑤確認メール送信  
- 代替シナリオ3a:在庫切れの場合、エラー表示して①に戻る  
- 事後条件:注文データが登録される

---

#### R-7. ユーザーストーリー

**目的**  
アジャイル開発において、利用者視点で要求を短く記述する。

**記載内容**  
書式:「<ロール>として、<目的>のために、<機能>したい」  
加えて受け入れ基準(Acceptance Criteria)。

**具体例**  
「経理担当として、月次決算を早期化するために、売掛金データをボタン1つでCSV出力したい。」  
受け入れ基準:出力時間3秒以内、文字コードUTF-8、ヘッダー行付き。

---

### 4.3 システム要件(System Requirements)レベル

#### R-8. 機能要件定義書(SRS: Software Requirements Specification)

**目的**  
IEEE 830 / ISO/IEC/IEEE 29148 に準拠し、システムが提供すべき機能を明確・検証可能な形で記述する。

**記載内容**  
- 機能一覧(機能IDを付与)
- 各機能の概要、入力、処理、出力
- 機能間の関係
- 優先度(MoSCoW法:Must/Should/Could/Won't)
- 業務ルール

**具体例**  
「F-001 ユーザー登録機能:利用者がメールアドレス、パスワード、氏名を入力すると、システムは入力チェック後に利用者マスタへ登録する。パスワードは8文字以上、英大小数字記号を各1文字以上含むこと。優先度:Must。」

---

#### R-9. 非機能要件定義書

**目的**  
機能以外の品質特性を定義する。ISO/IEC 25010(製品品質モデル)またはIPA非機能要求グレードを参照するのが一般的。

**記載内容(ISO/IEC 25010の8特性)**  
1. 機能適合性(Functional Suitability)
2. 性能効率性(Performance Efficiency)
3. 互換性(Compatibility)
4. 使用性(Usability)
5. 信頼性(Reliability)
6. セキュリティ(Security)
7. 保守性(Maintainability)
8. 移植性(Portability)

**具体例**  
- 信頼性:平日9-18時の稼働率99.9%以上、MTTR1時間以内、RPO15分、RTO4時間
- 性能効率性:ピーク時同時500ユーザー、画面応答95%tile以内に2秒
- セキュリティ:全通信TLS1.3以上、パスワードはbcrypt(cost=12)でハッシュ化、監査ログ7年保管

---

#### R-10. 外部インターフェース要件

**目的**  
他システムとの連携に関する要求を定義。

**記載内容**  
- 連携先システム一覧
- 連携目的・連携データ
- 連携方式(リアルタイム/バッチ、同期/非同期)
- 連携頻度、データ量
- エラー時の運用要件

**具体例**  
「顧客マスタをCRMシステムから取得する。差分データを毎朝5時にREST APIで取得。1日あたり平均1000件、ピーク5000件。連携失敗時は3回リトライ後に運用保守へメール通知。」

---

#### R-11. データ要件定義書

**目的**  
システムで扱うデータの種類、ボリューム、保管期間、権限などを定義。

**記載内容**  
- データエンティティ一覧
- データボリューム(初期、年間増加、5年後想定)
- 保管期間(法定保管期間含む)
- アクセス権限の方針
- 個人情報・機密情報の扱い

**具体例**  
「取引データ:初期100万件、年間増加500万件、5年後2500万件想定。法定保管期間7年。個人情報(顧客氏名・住所)を含むため、マスキング・暗号化必須。」

---

#### R-12. 制約条件一覧

**目的**  
プロジェクトに課せられた前提・制約を明示し、後の設計・実装判断の根拠とする。

**記載内容**  
- 技術的制約(既存システムとの連携、使用可能技術スタック)
- 法的制約(個人情報保護法、電子帳簿保存法、業界規制)
- 予算・スケジュール制約
- 組織的制約

**具体例**  
「基幹DBはOracle 19c固定(ライセンス契約上)」「電子帳簿保存法要件を満たすこと」「2027年4月本番稼働(法改正対応のため絶対厳守)」。

## 5. 基本設計(外部設計)フェーズ

利用者やシステム外部から見た振る舞いを設計します。発注者もレビューに参加する最後のフェーズ。

### 5.1 方式設計(アーキテクチャ設計)

#### B-1. システム方式設計書

**目的**  
システム全体をどう構成するか、ハードウェア・ソフトウェア・手作業にどう分担させるかを決定。

**記載内容**  
- システム全体構成(論理/物理)
- 機能の実現方式(自動化範囲、手作業範囲)
- ハードウェア構成
- ソフトウェア構成(OS、ミドルウェア、言語、フレームワーク)
- 冗長化・可用性方式
- 認証・認可方式
- ログ・監視方式

**目的**

**記載内容**

**具体例**

#### D-7. フローチャート・疑似コード(PDL)

**目的**

- 判断(ひし形)

```
  IF バックオーダー可 THEN

END IF
```

#### D-9. テーブル定義書・インデックス設計書

**目的**  
全テーブルの物理仕様とインデックスを確定。

**記載内容**  
- テーブル定義:物理名、論理名、カラム一覧(物理名、論理名、型、桁数、NULL可否、デフォルト、制約、説明)
- インデックス定義:索引名、対象カラム、ユニーク性、種類(B-Tree、Hash、GIN等)
- 想定行数、増加率、ストレージ見積もり

**具体例**  
```
t_order テーブル:
- order_id VARCHAR(20) PK
- customer_id VARCHAR(10) NOT NULL (FK)
- order_date DATE NOT NULL
- total_amount DECIMAL(12,0) NOT NULL DEFAULT 0
- status VARCHAR(20) NOT NULL
- created_at TIMESTAMP NOT NULL

インデックス:
- PK: order_id
- idx_customer_date (customer_id, order_date): 顧客別履歴検索用
- idx_status (status): ステータス別集計用
```

#### D-10. DDL

**目的**  
実際にDBに適用するCREATE文を定義。

**記載内容**  
- CREATE TABLE
- CREATE INDEX
- ALTER TABLE(制約、外部キー)
- CREATE VIEW, CREATE TRIGGER(必要に応じて)

**具体例**  
```sql
CREATE TABLE t_order (
  order_id     VARCHAR(20)   NOT NULL,
  customer_id  VARCHAR(10)   NOT NULL,
  order_date   DATE          NOT NULL,
  total_amount DECIMAL(12,0) NOT NULL DEFAULT 0,
  status       VARCHAR(20)   NOT NULL,
  created_at   TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (order_id),
  FOREIGN KEY (customer_id) REFERENCES t_customer(customer_id)
);
CREATE INDEX idx_customer_date ON t_order (customer_id, order_date);
```

- モジュール名
- 入力パラメータ(名前、型、意味、制約)
- 処理ロジック(疑似コード)
- 出力(戻り値、副作用)
- 使用するDB操作
- 例外
- 事前条件・事後条件

**具体例**  
```
モジュール名: placeOrder
入力: 
  - customerId: String (必須、顧客マスタに存在する値)
  - items: List<OrderItemDto> (必須、1件以上)
処理:
  1. 顧客存在チェック
  2. 全明細について在庫引当
  3. 金額計算(小計+税)
  4. 注文を永続化
  5. 確認メール送信(非同期)
出力:
  - Order エンティティ
例外:
  - CustomerNotFoundException
  - InsufficientStockException
事後条件: 注文データがDBに永続化されている
```

#### D-13. API詳細仕様書(OpenAPI等)

- リクエストボディ(JSON Schema)
- レスポンスボディ(ステータス別のスキーマ)
- エラーレスポンス一覧
- 認可ルール
- レート制限

**具体例**  
OpenAPI 3.0 YAML形式で記述:
```yaml
paths:
  /api/v1/orders:
    post:

      security: [{ bearerAuth: [] }]
      requestBody:
        content:
          application/json:
```

## 8. 取捨選択の指針

### 8.1 プロジェクト規模別の推奨セット

**小規模(~6ヶ月、5人以下)**  
- 必須:業務要件定義書、機能要件定義書(またはユーザーストーリー)、非機能要件定義書、用語集、画面設計書、ER図、テーブル定義書、API仕様書、単体テスト仕様書
- 省略可:RTM、配置図、コンポーネント図、アクティビティ図、状態遷移図(該当する場合のみ)

**中規模(6ヶ月~2年、10~30人)**  
上記に加えて:As-Is/To-Be業務フロー図、ユースケース図・記述、システム方式設計書、運用設計書、セキュリティ設計書、CRUD図、シーケンス図、クラス図、RTM

**大規模(2年以上、50人以上)**  
原則として全ドキュメント作成を検討。特に横断管理系(RTM、用語集、ステークホルダー一覧)と共通設計系(方式共通設計書、セキュリティ設計書、運用設計書)が重要。

### 8.2 開発手法別の傾向

**ウォーターフォール**  
- フェーズ毎に全ドキュメントを完成させてから次へ
- 設計書の網羅性・正確性が重視される
- 契約・検収の根拠となる

**アジャイル(Scrum等)**  
- ユーザーストーリーとインセプションデッキが中心
- 設計ドキュメントは「Just Enough」— 実装と同時並行で必要最小限
- コードとテストが一次ドキュメントとなり、補足として設計メモを残す
- ただしアーキテクチャ設計書、API仕様書、ER図は作成することが多い

### 8.3 判断基準

以下の問いで要否を判断すると良いでしょう。

1. **このドキュメントがないと誰が困るか?**  
   困る人が特定できないなら不要の可能性が高い。

2. **別のドキュメントや成果物(ソースコード、自動生成物)で代替できないか?**  
   例:テーブル定義書は、DDLと自動生成ツール(SchemaSpy等)があれば簡素化可能。

3. **将来の保守・拡張で参照されるか?**  
   一度読んで捨てられるだけなら軽量化を検討。

4. **監査・契約上必要か?**  
   金融、医療、公共系では監査証跡として必須の場合がある。

5. **ドキュメントのメンテナンスコストに見合う価値があるか?**  
   作ったきり更新されないドキュメントは却って害になる(誤った情報源になる)。

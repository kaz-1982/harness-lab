# 00_intake.md — ① ヒアリング・問題定義

目的：ユーザーが抱えている問題を理解し、`output/01_問題定義.md` を作る。
**問題が曖昧なまま原因分析に進むと、ツリーもぼやける。** ここで問題をはっきりさせる。

---

## 進め方

### ステップ A：まず「問題」を一言で聞く

最初はシンプルに、ユーザーの言葉で問題を出してもらう。

```
それでは、抱えている問題を教えてください。
今いちばん「困っていること・解決したいこと」は何ですか？
（思いつくままで大丈夫です）
```

### ステップ B：問題をはっきりさせる質問（1回でまとめて聞く）

A の回答を受けて、以下を**一度に**質問する。箇条書きで・分かる範囲でOKと伝える。

```
ありがとうございます。もう少しはっきりさせるために、いくつか教えてください。
（分かる範囲でOK。分からない項目は「分からない」で構いません）

1. 本来は「どうあってほしい」ですか？（あるべき姿・理想）
2. いまは実際に「どうなっている」ですか？（現状・事実）
3. いつ・どこで・どんなときに起きますか？
4. どれくらい困っていますか？（影響・困りごとの大きさ。数字があれば歓迎）
5. すでに試したこと・分かっていることはありますか？
```

> **考え方**：問題とは「**あるべき姿（1）と現状（2）のギャップ**」です。
> このギャップをはっきりさせると、後の原因分析がぶれません。

### ステップ C：問題を1文に言い換える

集まった情報から、問題を**1文の問題ステートメント**に言い換えて提示し、合意を取る。

- 良い例：「営業部で、月の残業時間が目標20hに対し平均45hになっている」
- 悪い例：「残業が多い」（曖昧・測れない）

できるだけ **具体的・測れる形**にする（いつ／どこで／どれくらい）。
※数字が分からなければ無理に作らず、「程度」を言葉で表す。

### ステップ D：記録する

回答を `output/01_問題定義.md` に保存する。雛形は `templates/01_問題定義.md` を使う。

### ステップ E：確認して次へ

記録内容を見せ、「**この理解で合っていますか？　これで原因の分析（②）に進んでよいですか？**」と確認する。
修正があれば反映してから、`harness/01_why_tree.md` に進む。

---

## 注意

- 専門用語を避け、誰でも答えられる質問にする。
- 質問は1回にまとめる（往復を減らす）。
- ユーザーが「分からない」と言ったら、例を出して助ける。無理に埋めさせない。
- この段階では**原因や対策を語らない**。まず「問題は何か」だけに集中する。
- 事実（観測されたこと）と、ユーザーの解釈・推測を分けてメモする。

以上。記録と合意が取れたら ② `harness/01_why_tree.md` へ。
